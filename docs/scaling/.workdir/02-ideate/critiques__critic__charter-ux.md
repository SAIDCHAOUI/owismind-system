# ADVERSARIAL CRITIQUE - Charter + Trust-UX + Accessibility lens (with cross-cutting factual corrections)

## BLOCKER

**B1 - "Client-side CSV/PNG from rows already in the store" is FALSE; the captured result is a 200-row PROOF EXCERPT, not an export source.** `arch:webapp-capabilities` #2 says "the result rows are already in the `evidence` store... serialize to CSV in JS." So **every CSV/XLSX/"raw numbers" export claim across `arch:webapp-capabilities` (#2, #9) and `arch:deliverables` (Phase 0b, 1) ships a silently-truncated 200-row file** to an account manager who thinks they got the full account list. This is a trust anti-pattern (silent data loss) AND a correctness bug. The plans MUST either: caveat "export = the proof excerpt, capped at 200 rows, with a visible truncation banner," or design a separate bounded re-query path (which then collides with non-negotiable #3 - no SQL re-run). They cannot claim "the raw numbers behind the answer" without this caveat.

**B2 - The factory's `get_capabilities()` runtime overlay does not exist and contradicts the frozen-paste constraint.** `arch:agent-factory` §1 asserts the orchestrator "already reads via `dataiku.api_client`, so a read of `webapp_settings_v1` is safe... extend `get_capabilities()` to overlay the operator JSON onto the static literal at load." Adding a `webapp_settings_v1` read into a **standalone Code Agent file that cannot import the plugin** (the proposal's own frozen constraint) means hand-duplicating the SQL/identity/cap discipline into the pasted agent, on every agent's hot path, at load. That is a security-sensitive net-new surface (rule #3: parametrized + bounded + read-only inside agent code with no shared lib), not "a day." The factory's central "single source of truth with operator overlay" mechanism is **NEEDS-DSS-VALIDATION and architecturally heavier than claimed** - downgrade it or drop the runtime overlay in favor of pure build-time codegen.
