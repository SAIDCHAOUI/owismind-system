# WEBAPP TOOLS USABLE BY ANY AGENT (incl. foreign/visual) - Senior Architect Brief

## 1. Diagnosis: why chart/table are intrinsic today (the 4-layer coupling)

`show_chart` / `show_table` / `show_kpi` are **not tools** in any portable sense. They are Python function specs built inside `build_tool_specs(caps)` (`OWIsMind_orchestrator.py:285`) and dispatched **inline in `node_tools`** (line 1653: `if name in ("show_chart","show_table","show_kpi")`). They render nothing themselves - they read `state["latest"]` (line 1470: `latest = state.get("latest")`), which was populated only because **our own** `_run_subagents` captured the sub-agent's SQL result and set `updates["latest"] = result` (line 1649). The visible chart then needs four things that only exist inside our stack:

1. **Shared in-memory state** - `state["latest"]` = `{columns, rows}`, mutated by our LangGraph node. A foreign agent has no such state.
2. **The LangGraph custom stream** - the spec is emitted as an `ARTIFACT` event on `get_stream_writer`; `_normalized_artifact_event` (`streaming.py:142`) is the *only* reader, gated on `eventKind == "ARTIFACT"` (line 400) and `_ARTIFACT_CHART_TYPES = ("line","bar","pie")`.
3. **Our backend stream interpreter** - `stream_manager._worker` accumulates specs (`stream_manager.py:390`, cap `MAX_ARTIFACTS_ACCUM=8`) and on DONE calls `artifacts_storage.save_artifacts(exchange_id, user_id, artifacts)` (line 454). **No `exchange_id` → nowhere to land.**
4. **The data join** - the spec stored in `webapp_artifacts_v1` holds *no data*; `/evidence/meta` re-joins it with `generated_sql[].result` we captured. A foreign agent's rows never entered `generated_sql`.

So the honest conclusion: **a foreign/visual agent run outside our LangGraph code cannot populate the Evidence panel today.** Not a permissions gap - a *plumbing* gap. The render verbs are coupling glue, not transferable capabilities. This is the crux of the user's question, and the answer is: chart/table-into-side-panel is the *hardest* thing to externalize; PDF/email/CSV are the *easiest*.

## 2. TRACK A - Plugin agent tools (the user's instinct, correct)

Dataiku ships a first-class path: **plugin-packaged custom agent tools** (`python-agent-tools/<tool>/tool.json` + `tool.py` extending `BaseAgentTool`). Our `attribute_lookup_tool.py` already implements that exact contract (`get_descriptor` → `{description, inputSchema}` at line 392; `invoke(input, trace)` → `{"output": ..., "sources":[...]}` at line 458) - it is just a hand-created project object, **not yet plugin-packaged**. Plugin-packaging is what makes a tool appear in the GenAI catalog and be **selectable by visual AND code agents**. That instinct is right; the discipline is to split tools into two classes by whether they need our side panel.

**Class 1 - self-contained tools (VERIFIED-feasible, build now):** the tool input carries everything, the tool output *is* the deliverable. No `exchange_id`, no state, no panel.

- **`build_pdf`** - *(a)* a plugin tool that fills a FIXED versioned template with model-supplied JSON slots and returns a download URL. *(b)* **AM**: leave-behind client one-pager; **PO**: product perf sheet; **marketing dir / exec**: branded board-ready summary. *(c)* `tool.json` descriptor + `tool.py` `invoke()`: `inputSchema` = `{title, sections:[{heading, paragraphs[], table?{columns,rows}, chart_png_b64?}]}` (model fills *slots only*, never structure - the safety property). Render with **ReportLab** (pure-Python, pip-only, NO system libs - unlike WeasyPrint which needs pango/cairo). Write the PDF to a managed folder via `dataiku.Folder`, return `{"output": json.dumps({"url": signed_path, "pages": n})}`. *(d)* **M, needs user install of ReportLab into the plugin code env** (flag explicitly: non-negotiable #1). *(e)* Charter applies only to on-screen UI, not the PDF, but reuse `#FF7900`/Helvetica for brand consistency; instance-safe (no DB, bounded page count). *(f)* depends on a code env + managed folder; Phase 2.
- **`draft_email`** - *(a)* fills a fixed email template → returns `{subject, body_html, body_text}`; **never sends from the tool**. *(b)* **AM**: renewal/QBR follow-up draft; **exec**: exec-summary mail. *(c)* `invoke()` returns the JSON in `{"output": ...}`; **HITL preview in the webapp before any send**; `smtplib` send lives behind a *separate* `POST /owismind-api/email/send` with `requires_confirmation`, never inside the tool. *(d)* **S, zero install** (smtplib + template are stdlib). *(e)* security: blast radius - gate behind explicit confirm + audit log; charter N/A. *(f)* Phase 2, independent.
- **`query_<domain>`** - *(a)* a thin plugin wrapper exposing a domain's semantic query as a normal selectable tool. *(b)* lets a *foreign* visual agent answer "revenue of X" without our orchestrator. *(c)* `invoke()` internally does `get_default_project().get_agent_tool("v4oqA6R").run(payload)` - exactly how the sub-agent already calls it; returns rows as a JSON string. *(d)* **M, no install.** *(e)* read-only, timeout - instance-safe; ensure the tool runs *as the calling user* (access governance). *(f)* Phase 3, after Track B.

**Class 2 - `render_chart` / `render_table` (NEEDS-DSS-VALIDATION, the hard one):** *(a)* a plugin tool that takes serialized rows + a chart spec and lands them in `webapp_artifacts_v1` so the Evidence panel shows them, even for a foreign agent. *(c)* the input MUST carry the data (the foreign agent has no `state["latest"]`): `inputSchema` = `{kind:"chart"|"table", chart_type?, x?, y?, columns, rows, exchange_id?}`. `invoke()` would: validate/cap (reuse the *exact* shape `artifacts_storage._sanitize` already enforces - `MAX_ARTIFACTS_JSON_CHARS=16_000`, kinds `("chart","table","kpi")`), then **if an `exchange_id` is present**, write both the spec AND the rows so `/evidence/meta` can render without a `generated_sql` join (this is the one schema change: artifacts must optionally carry inline `data`, not only a spec - a *new* `webapp_artifacts_v2` with a `data` column, no-ALTER per non-negotiable #6, or a parallel `webapp_tool_artifacts_v1`). *(d)* **L, no install but real plumbing + a spike.** *(e)* THE COUPLING SOLUTION: this **only works when the agent is invoked from OUR webapp session**, because that is the only context where an `exchange_id` exists and is owner-scoped. When a user runs a visual agent *inside a Dataiku chat* (not our webapp), there is no `exchange_id` and no panel - the tool must **degrade gracefully** to returning the table as JSON in `{"output": ...}` (still useful, just no side panel). Be brutally honest with the user here: **the side panel is a property of our webapp session, not of the tool.** A tool can write to it *iff* our session passed an `exchange_id` through.

The migration that makes this safe: keep the built-in `show_chart`/`show_table` **exactly as-is** for our orchestrator (don't touch the working flow), and add `render_chart` as a *new* plugin tool that other agents select. Our orchestrator could *later* be switched from built-in to the plugin tool, but only after the plugin tool proves equivalent - never as a big-bang.

## 3. TRACK B - Backend as tool caller (VERIFIED, cheapest leverage, build first)

The Flask 3.9 backend can call **any managed tool** via `dataiku.api_client().get_default_project().get_agent_tool(id).run(payload)` - the orchestrator already does this for `attribute_lookup` and `v4oqA6R` (`OWIsMind_orchestrator.py:1293`), proving the API works from our process. *(a)* a small server module `owismind/agents/tool_caller.py` wrapping `get_agent_tool(id).run()` with timeout + owner context. *(b)* **all personas**: unlocks deterministic, *agent-free* features - a "Download PDF of this answer" button that calls `build_pdf` directly from `/owismind-api/export/pdf`, no LLM in the loop, no quota burn, no hallucination risk. *(c)* sits beside `evidence/service.py`; reuses the captured `generated_sql[].result` as the PDF/CSV payload. *(d)* **S, no install** (the call API is proven). *(e)* instance-safe (sync, bounded), and it sidesteps the whole foreign-agent question for the 80% case: *most* deliverables (PDF/CSV/PNG) don't need an agent at all - the **frontend already holds the Chart.js instance** (`ArtifactChart.vue`), so **chart PNG export is pure client-side `chart.toBase64Image()` + `a[download].click()`, ZERO install, ship this week** as the unwired download icon's handler. *(f)* foundational; everything in Track A's Class 1 becomes trivially reachable through it.

## 4. TRACK C - MCP server (FUTURE, do not build now)

*(a)* expose OWIsMind tools over MCP for cross-product reuse. *(b)* org-wide: other teams' agents query revenue. *(c)* needs **DSS 14.0+ and Python 3.10+** - our backend is 3.9.23 (non-negotiable #8), so this cannot host on the current backend without a separate 3.11 service. *(d)* **L, infra-dependent, NEEDS-DSS-VALIDATION of DSS version.** *(e)* new attack surface + auth - out of scope until Tracks A/B prove demand. *(f)* **YAGNI now.** Park it.

## 5. RECOMMENDATION - what to build, what to spike first, migration path

**Spike FIRST on the instance (one afternoon, blocks the architecture bet) - these are NEEDS-DSS-VALIDATION:**
1. Package `attribute_lookup` as a `python-agent-tools/` plugin tool, install it, and **confirm a VISUAL agent can select it** (this single test validates the entire Track A premise).
2. Confirm the **plugin-tool code env runs at 3.9** (or which version it forces) - determines whether ReportLab/openpyxl availability matches.
3. Prove the **`exchange_id` handoff**: from our webapp session, pass `exchange_id` into a tool input and confirm `invoke()` can write an owner-scoped row our `/evidence/meta` reads. If this fails, `render_chart`-for-foreign-agents is dead and you fall back to "tool returns JSON, no panel" - know this *before* building.

**Build NOW (VERIFIED-feasible, in order):**
- **Phase 0 (days):** wire the existing download icon → client-side chart PNG (`toBase64Image`) + CSV via stdlib `csv`/`io` from the captured result. No install, no agent, no risk. Immediate persona value.
- **Phase 1 (Track B):** `tool_caller.py` + `/owismind-api/export/pdf` calling a `build_pdf` plugin tool. Requires user to install **ReportLab** into the plugin code env - *state this in the plan*. Agent-free, deterministic.
- **Phase 2 (Track A Class 1):** package `build_pdf`, `draft_email` as plugin tools so agents can *also* invoke them. `draft_email` strictly draft-only + HITL send route + audit (`webapp_audit_v1`).
- **Phase 3 (Track A Class 2):** `render_chart`/`render_table` plugin tools **only if the exchange_id spike passed**, with graceful JSON degradation otherwise. Keep `show_chart` built-ins untouched throughout; never big-bang the working chart flow.

**Migration discipline:** built-ins and plugin tools coexist; the plugin tool is *additive*. Freeze the `ARTIFACT`/`artifacts_storage` contracts (non-negotiable: frozen event kinds); any inline-data need is a *new* `_v2` table, never an ALTER. The honest one-liner for the user: **PDF/email/PNG/CSV externalize cleanly today via Track B; charts-into-the-side-panel for a foreign agent is gated on the exchange_id spike - validate it before betting on it.**
