# ADVERSARIAL CRITIQUE - LENS: SECURITY & GOVERNANCE

I verified the spine the proposals lean on: `resolve_identity(request.headers)` (never from body), `_evidence_guard()` (routes.py:639), `save_artifacts(exchange_id, user_id, …)` with `ON CONFLICT (exchange_id) DO UPDATE` (artifacts.py:101-123), `read_artifacts` owner-scoped `WHERE exchange_id AND user_id` (artifacts.py:140-151), per-user throttle. Confirmed ABSENT today: any SMTP/email, any audit table, any PII masking (the only `mask` hits are SQL-comment masking, unrelated). The proposals are mostly security-aware, but several claims are over-confident and one is an outright vulnerability waiting to be built.

## BLOCKER

**B1. The artifact UPSERT is keyed on `exchange_id` alone - a forged `exchange_id` from a foreign-agent tool would let one user OVERWRITE another user's artifacts.** `tool-exposure` Class 2 (`render_chart`) and `deliverables` Phase 4 both propose passing `exchange_id` *into a plugin tool's `invoke()`* so it writes `webapp_artifacts_v1`. But `save_artifacts` does `ON CONFLICT (exchange_id) DO UPDATE SET artifacts=…, user_id=EXCLUDED.user_id` (artifacts.py:121-123) - it trusts whatever `user_id` is handed in and the PK is `exchange_id` only. A plugin tool's `invoke(input, trace)` receives input **the model/agent controls**; `exchange_id` would be an attacker-influenceable field. If a foreign agent (or a crafted prompt) supplies another user's `exchange_id`, it can clobber that row AND reassign `user_id`, i.e. cross-tenant artifact write + ownership takeover. `read_artifacts` being owner-scoped does NOT save you - the WRITE path has no owner check. **Required control before any tool-writes-artifacts design:** the write must (a) never accept `user_id`/`exchange_id` from tool input; the backend must mint the binding from the authenticated webapp session, and (b) the UPSERT must be guarded `WHERE existing.user_id = :session_user` (reject, not silently reassign, on owner mismatch). `tool-exposure` is honest that this is `NEEDS-DSS-VALIDATION`, but it underweights that *even if the exchange_id handoff technically works, the current save path is exploitable* - the spike must validate the **owner-binding**, not just the plumbing. Until then, treat "tool writes to Evidence panel" as a known-unsafe pattern, not merely unvalidated.

## HIGH

**H2. Export endpoints (`/evidence/export`, `/report`) inherit IDOR risk if they don't route through `_evidence_guard` exactly.** `deliverables` and `webapp-capabilities` both add `GET /evidence/export?exchange_id=…`. `deliverables` says "reuse the shared guard (routes.py:640)" - correct instinct, but the guard returns a tuple `(identity, err)` and the load-of-captured-result must filter `WHERE exchange_id AND user_id` like `read_artifacts`. None of the three export-proposing docs state the WHERE explicitly; `webapp-capabilities` #2 even floats a *client-side* CSV path "the rows are already in the evidence store" - fine, but the moment a server path exists it MUST owner-scope and MUST NOT accept a table/connection/column from the query (only `exchange_id` + an enum `format`). **Required:** every export route = `_evidence_guard()` + owner-scoped read of stored JSON + `format ∈ {csv,xlsx,pdf}` validated against a server enum + `MAX_EXPORT_ROWS` cap. State the WHERE in the plan, not just "owner-scoped" in prose.

**H3. Filename / Content-Disposition injection is unaddressed in all export proposals.** `webapp-capabilities` #1 derives the PNG filename from "conversation title + chart label"; `deliverables` builds `Content-Disposition: attachment; filename="..."` from a slug. Conversation titles and chart labels are **user/model-controlled strings**. Unsanitized, they enable header injection (CRLF in `Content-Disposition`) and reflected-download naming tricks. **Required control:** never interpolate user/model text into the header; whitelist `[A-Za-z0-9_-]`, hard-cap length, and use a server-generated base name (e.g. `owismind_<exchange_id>.csv`). This is a concrete injection vector the term "filename" in two proposals waves past.

**H4. Email is the highest-blast-radius surface and the proposals are INCONSISTENT about whether send is even in scope.** Good: all three email-touching docs (`deliverables` Phase 3, `tool-exposure` `draft_email`, `webapp-capabilities` #13) mandate draft-only + HITL + `requires_confirmation`. But `webapp-capabilities` #13 ("KPI digest pushed to email") proposes **scheduled, unattended `smtplib` send** - that is auto-send with NO human in the loop per message, directly contradicting the HITL discipline the other proposals correctly insist on. A scheduled digest that emails an un-reconciled (possibly hallucinated) revenue number to a client is exactly the "permanent disengagement" red line, now automated and un-gated. **Required:** (a) NO server-side send in v1, period; (b) if send ever ships, it is per-message HITL with a server-enforced `requires_confirmation` gate (front sends opaque intent, backend gates - NON-NEGOTIABLE #4), recipient on a server-side allowlist (never a free-text `to` from the model - anti-exfiltration), SMTP creds in admin config like `sql_config` (never hardcoded), and rate-limited + audited; (c) the scheduled digest must be reclassified as draft-to-an-internal-inbox or dropped, not auto-send-to-clients. The `draft_email` slot schema in `deliverables` has `to_hint` (good - a hint, not an address) but `tool-exposure`'s `query_<domain>` "runs as the calling user" claim and the digest's auto-send are in tension; resolve loudly.

**H5. PII in deliverables and the new Evidence Dataset tab is governance-MISSING, asserted but not designed.** `domains-and-360` and `webapp-capabilities` #10 both say "PII masking in the Dataset tab" / "mask before render" as a one-liner, with NO mechanism. The captured `generated_sql[].result` can contain customer names, carrier codes, account managers, contacts. An export (CSV/XLSX/PDF) is a **durable, forwardable** copy of that PII leaving the governed app - far higher exposure than an on-screen drill. None of the proposals define WHO may export PII, WHAT is masked, or that exports are logged. **Required, and currently absent from every proposal as an actual design:** a PII policy (column-tag-driven masking or an admin-gated "export-with-PII" entitlement), applied at the export-serialization point, plus mandatory audit of every export (who exported which exchange, row count, format). Calling it out in prose is not a control.

## MEDIUM

**M6. Audit log is proposed but its COMPLETENESS and tamper-resistance are not specified, and it is sequenced too late in two docs.** `domains-and-360` A2 (`webapp_audit_v1`) and the verified governance research both want append-only run/cost/tool/sql/quota provenance - good, and it correctly mirrors `storage/usage.py` discipline. But: (a) `webapp-capabilities` and `deliverables` add export/email/tool surfaces with **no audit of those actions** - an export route and an email draft are exactly what you must log for governance, yet only `domains-and-360` mentions audit and scopes it to agent runs. **Audit must cover exports, draft/send, and tool invocations, not just chat runs.** (b) "Append-only" is asserted but nothing enforces it (no REVOKE UPDATE/DELETE, no hash chain) - at minimum document that the table is write-once by app convention and admin-readable only. (c) Audit should be a Phase-A prerequisite across ALL three missions, not a single-doc item.

**M7. Quota bypass via the new agent-free / tool-caller paths.** `tool-exposure` Track B advertises "a Download PDF button that calls `build_pdf` directly, **no LLM in the loop, no quota burn**" as a *feature*. But `tool-exposure` `query_<domain>` and the agent-callable `build_pdf` CAN invoke LLM-backed semantic tools (the semantic tool runs Sonnet). If those server-side tool calls bypass `budget.has_budget()` (enforced today only in `/chat/start`, routes.py:315), a user past their $50 cap can keep spending via export/tool routes. **Required:** any route that can trigger an LLM-backed tool call must pass the same `budget.has_budget(user_id)` gate as `/chat/start`; deterministic-only routes (CSV/PNG/pure-template PDF over already-captured results) are correctly quota-free, but the plan must DISTINGUISH them and gate the LLM-touching ones. The "no quota burn" framing is dangerous if generalized.

**M8. The semantic cache (`domains-and-360` B1, `webapp-capabilities` #14) is a cross-user data-leak risk if the key isn't user/entity-scoped.** Caching `(question+resolved_filters)->(result+narrative)` and serving it to "instant repeat answers" must NOT serve user A's account-scoped result to user B. If two users ask about different accounts the resolved filters differ (safe), but a shared cache of business results still crosses the owner-scoping boundary that every other read enforces. **Required:** cache key includes the resolution context and the cache read is owner/entitlement-aware, OR the cache only stores non-row-level aggregates. `domains-and-360` keys on "resolved filters" (better) but does not state the leak constraint; make it explicit.

**M9. Reconciliation is correctly P0 but is a DETECTOR, not a GATE - and no proposal says what happens when it fails.** All three governance-aware docs add claim-vs-result reconciliation (good, cheap, 3.9-safe). But flagging "unverified" while still rendering the number to an executive is weak: the red line is one hallucinated number seen once. **Required governance decision (missing):** for deliverables/email specifically, an unreconciled number should BLOCK the export/draft (abstention > hallucination, which the research itself states), not merely badge it. A flagged number in a forwardable PDF is the worst case.

## LOW

**L10. `tool-exposure` "run as the calling user" for `query_<domain>` is asserted, not verified** - whether `get_agent_tool().run()` from the Flask process carries the caller's DSS access context (vs the webapp service identity) is `NEEDS-DSS-VALIDATION`; if it runs as the service account, dataset-level access governance is bypassed. **L11.** Plugin-tool input schemas (`build_pdf` `sections[]`, `render_chart` `rows`) accept model-supplied bytes/`chart_png_b64` - base64 image slots are an unbounded-size / content-type vector; cap size and validate it's a PNG, don't embed arbitrary base64 into a PDF. **L12.** `agent-factory`'s `enabled_agents` overlay correctly keeps `allowed_tables`/`lookup_dataset` dev-owned in `registry.json` (good - honors #3/#4); just ensure the admin meta-edit path (`validate_agent_meta`) can NEVER reach `agent_id`/`semantic_model_id` even via a crafted JSON body.

## TOP 5 THINGS THE FINAL PLANS MUST CHANGE OR CAVEAT

1. **Fix the artifact WRITE path before any tool-writes-Evidence design (B1):** PK is `exchange_id`-only and the UPSERT reassigns `user_id` from input - guard the write with `WHERE existing.user_id = session_user`, mint the binding server-side, never accept `user_id`/`exchange_id` from tool input. This is a real cross-tenant write/takeover, not a future worry.
2. **Every export route: `_evidence_guard` + explicit `WHERE exchange_id AND user_id` + `format` enum + `MAX_EXPORT_ROWS` + sanitized server-generated filename (H2/H3).** No table/column/connection ever from the query.
3. **NO server-side email send in v1; reclassify the auto-send KPI digest as draft-only (H4).** If send ever ships: per-message HITL `requires_confirmation` gated server-side, recipient allowlist (no model-chosen `to`), admin SMTP creds, rate-limit + audit.
4. **PII export governance is a missing DESIGN, not a one-liner (H5):** define masking/entitlement at the serialization point and audit every export; exports are durable PII copies leaving the app.
5. **Audit must span exports/email/tool-calls (not just runs) and be a cross-mission Phase-A prerequisite; and any LLM-touching export/tool route must re-apply `budget.has_budget()` (M6/M7).** Make reconciliation a GATE (block unverified numbers) for deliverables, not just a badge (M9).
