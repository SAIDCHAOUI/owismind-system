# ADVERSARIAL CRITIQUE - LENS: DATAIKU INSTANCE SAFETY (non-negotiable #2)

Verified against real code this session. The five proposals are competent and mostly honor the obvious caps, but several "ship now, zero risk" claims hide instance-load failure modes, and one whole class of feature multiplies an existing per-run cap in a way nobody modeled.

## BLOCKER

**B1. The 360 fan-out is 8x3 = 24 concurrent semantic-tool/Sonnet runs, not 3.** `arch:domains-and-360` says "fan-out is already capped at 3" and "Do NOT raise MAX_PARALLEL_AGENTS" - but it reads only `MAX_PARALLEL_AGENTS=3` (orchestrator:141) and ignores `MAX_CONCURRENT_RUNS=8` (stream_manager.py:46). These are **multiplicative**: 8 webapp runs each fanning out 3 sub-agents = **24 simultaneous sub-agent threads**, each issuing 1-2 calls to the semantic tool, which runs **Sonnet + live SQL on SQL_owi**. That is 24 concurrent Sonnet calls + up to ~48 concurrent PostgreSQL queries on one shared connection, from a mono-process Flask backend with a small worker pool. Under a demo where several account managers each ask for a 360, this saturates the LLM Mesh budget AND the SQL connection simultaneously. The plan MUST add: a **global semaphore on concurrent sub-agent invocations** (e.g. cap total in-flight sub-agents across all runs, not per-run), and treat a 360 as consuming N run-slots, not 1. Without this, the headline feature is the instance-DoS feature.

**B2. value_index and value_catalog recipes have NO row cap - full table into pandas RAM, unguarded.** Every "add a domain" path (`arch:agent-factory` §5 step 1, `arch:domains-and-360` (1)) says "run the three recipes." Verified: `build_value_index_recipe.py:132` and `build_value_catalog_recipe.py:221` do `get_dataframe(infer_with_pandas=False)` with **no `MAX_ROWS` guard at all** (only the profiler caps at 2M, profile_dataset_recipe.py:50/554 - and even that loads the entire frame *before* `.head()`). For the revenue dataset (~175k rows) this is fine; the factory's whole premise is **onboarding bigger/unknown datasets** (tickets, billing). A multi-million-row ticket export run on-demand from a non-dev's "add agent" action = OOM on a shared DSS node. The plans MUST: (a) cap all three recipes uniformly and load via `iter_dataframes`/chunked or a `LIMIT`-pushed SQL read, not a single `get_dataframe`; (b) make recipe runs **scheduled off-peak scenarios only**, never triggered from the webapp/admin UI; (c) state a max-dataset-size precondition per domain in the onboarding playbook.

## HIGH

**H1. Synchronous PDF/XLSX render pins a worker thread in a mono-process backend.** `arch:deliverables` Phase 2 says PDF renders "synchronous in the request thread (acceptable)... renders in well under a second... No background job system needed (YAGNI)"; `arch:tool-exposure` repeats "well under a second." Unverified and risky. The backend is explicitly a "mono-process polling backend" (routes.py:661) whose existing throttle exists precisely so "a scripted flood must not pin worker threads." ReportLab+matplotlib Agg are CPU-real; matplotlib import alone is ~1s cold, and Agg figure render + ReportLab compose under contention is unpredictable. With a small worker pool already serving 500ms `/chat/poll` loops for every active conversation, a handful of concurrent PDF requests starves polling for ALL users (chat appears frozen). The plans MUST: (a) put `/export` and `/report` behind a **dedicated, stricter token bucket** (the existing `evidence_throttle` does NOT cover new routes - it is keyed only inside `_evidence_guard`); (b) add a **hard concurrency semaphore** (e.g. max 2 in-flight renders process-wide) returning 429 beyond it; (c) hard-cap `MAX_PDF_CHARTS`/embedded rows; (d) actually measure render time on the instance before asserting "<1s."

**H2. Export reads the 200-row PROOF EXCERPT, not the real result - silent truncation sold as "raw data."** Both deliverables proposals sell CSV/XLSX as "hand a client the raw numbers" / "pull rows into Excel." But the captured result is hard-capped at **MAX_RESULT_ROWS=200, MAX_RESULT_COLS=50, MAX_RESULT_JSON_CHARS=100k** (capture.py:33-38) - it is documented as "a PROOF EXCERPT, not a data export." So "Export CSV" silently hands the user a 200-row slice of a 50k-row answer with no indication it was truncated. This is both a correctness/trust bug (the user's red line is data they cannot defend) AND, if "fix" by re-running the SQL to get full data, an **instance-load bug** (re-executing agent SQL on export, violating both proposals' own "NEVER re-run agent SQL" rule and #3). The plans MUST: keep export bound to the captured excerpt, **label it explicitly as a capped excerpt in the file/UI**, and DROP the "raw numbers for the client" persona claim - or design a separate, governed, paginated, off-peak full-export path and call it what it is (a new feature, not a Phase-0 freebie).

**H3. Semantic cache is an unbounded process-memory growth + correctness hazard.** `arch:webapp-capabilities` #14 and `arch:domains-and-360` B1 propose an in-process or table cache keyed on `(question+filters)->(result+narrative)`, called before every sub-agent. In a mono-process backend an in-process dict grows unbounded across the process lifetime (no eviction named) = slow memory leak on a long-lived worker. The table variant adds a read+write on the hot `/chat/start` path. And **staleness on a shared analytics DB is a correctness landmine**: if DRIVE_Revenues is refreshed, a 24h-TTL cache serves stale revenue - the exact "wrong number" disengagement risk. The plans MUST: bound the cache (LRU with explicit max entries + TTL), key it so a dataset-refresh timestamp invalidates it, disclose "from cache" in the UI, and gate it behind real repeat-rate metrics (the model cascade already routes 85-90% cheap, so the ROI is unproven). `arch:webapp-capabilities` at least flags YAGNI; `arch:domains-and-360` wrongly elevates it to a P0 prerequisite.

## MEDIUM

**M1. Scheduled KPI digest (`arch:webapp-capabilities` #13) scanning "many KPIs on a cron" is a quiet full-scan multiplier.** Each digest re-runs the agent (LLM + SQL) per user per KPI. For N users x K KPIs this is a burst of agent runs on a schedule. The proposal correctly says "off-peak scheduled scenario, never a webapp cron" - good - but MUST add: a single batched scenario (not per-user runs), a cap on total digest runs, and reuse of cached/precomputed aggregates rather than one live agent call per user-KPI. As written ("reusing the agent to produce a digest") it is a fan-out bomb on a timer.

**M2. AI conversation title fires a Flash-Lite call inside `start_run` (`arch:webapp-capabilities` #4).** Low individual cost, but it adds an LLM call + a DB write to **every conversation's first exchange completion**, inside the already-bounded worker. Under load this competes with the chat answer itself for the worker's wall-clock (MAX_RUN_SECONDS=300 is shared). MUST: fire title generation **after** the answer is persisted/streamed (never blocking the answer), make it strictly best-effort/fire-and-forget, and skip it entirely if the run is near its time budget.

**M3. matplotlib presence is asserted, not verified, and import cost is ignored.** `arch:deliverables` says Agg is "almost certainly already in the DSS env"; `arch:tool-exposure` ships ReportLab with a user-install flag but is vaguer on matplotlib. Per #1 (NO INSTALL) and #8 (no unproven env claims), the plans MUST mark matplotlib **and** openpyxl as NEEDS-DSS-VALIDATION before any phase that depends on them, and account for first-call import latency (a cold matplotlib import inside a synchronous render thread compounds H1).

**M4. The "exchange_id handoff" spike (`arch:tool-exposure` Class 2) could let a foreign agent write our owner-scoped artifacts table - validate the AUTHZ, not just the plumbing.** The spike as framed asks "can a tool write a row our /evidence/meta reads." Instance-safety-adjacent security: if a plugin tool accepts an `exchange_id` as input, a foreign agent (or a crafted invocation) could pass **someone else's** exchange_id and inject artifacts into their panel, unless the write re-derives ownership server-side from the calling identity (not from the tool input). The plan MUST specify that the artifact write is owner-scoped by the **authenticated session**, never by a tool-supplied exchange_id. `arch:tool-exposure` correctly flags this NEEDS-DSS-VALIDATION but frames it as feasibility, not authorization.

## LOW

**L1.** `arch:agent-factory` Path A ("packaged `owismind_specialist` lib in the 3.11 env") is correctly deferred, but note it also adds a shared dependency whose version drift could break all pasted agents at once - a reliability (not just install) cost; Path B (codegen) is the right call.

**L2.** `arch:domains-and-360` "delivery is multi-table -> needs a pre-flattened dataset upstream" - that pre-flattening recipe is itself another full-table pandas join with no cap mentioned (same B2 class). Flag it.

**L3.** New `webapp_audit_v1` append-only insert at every terminal event (`arch:domains-and-360` A2) is cheap but unbounded growth over time; needs a retention/rotation note so the table doesn't become an instance storage problem at scale.

## TOP 5 THINGS THE FINAL PLANS MUST CHANGE OR CAVEAT

1. **Model the 8x3=24 concurrency, not 3.** Add a process-wide semaphore on total in-flight sub-agents and count a 360 as multiple run-slots. (B1 - the single sharpest instance-safety hole; the flagship feature is the DoS.)
2. **Cap and off-peak-schedule ALL three recipes; never trigger recipe runs from the webapp/admin UI; state a max-dataset-size precondition per domain.** value_index/value_catalog have zero row cap today. (B2)
3. **Put every new render/export route behind its own stricter throttle + a process-wide render semaphore (max ~2), and measure render time on the instance before claiming "<1s."** The existing `evidence_throttle` does not cover new routes. (H1)
4. **Export = the 200-row capped PROOF EXCERPT; label it as such and drop the "raw numbers for the client" promise.** Never re-run agent SQL to "fix" it. (H2)
5. **Cache (if built at all) must be bounded + dataset-refresh-invalidated + "from cache"-disclosed, and demoted from P0 to metrics-gated.** Stale revenue is the user's red-line failure. (H3)
