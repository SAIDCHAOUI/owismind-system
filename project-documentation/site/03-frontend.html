Overview and structure

The OWIsMind frontend is a Vue 3 + Vite SPA that lives under Plugin/owismind/frontend/. It is compiled once into hashed static bundles served by Dataiku DSS from the plugin resource folder. There is no Node server at runtime.

NO INSTALL - Project rule #1

The agent never runs npm install, npm ci, or any package manager command. Only the user installs. This is enforced by harness deny-list and the guardrail.sh pre-tool hook. Tests use the native node:test runner (no Vitest installed).

Source tree

Bootstrap: main.js sequence

The order of operations in src/main.js is load-bearing. Reversing any step breaks the app in production.

1. Import styles in order

tokens.css THEN base.css. The order is required because base.css consumes the CSS variables defined by tokens.css. Reversing them yields unresolved values at first paint.

2. Set theme on <body> BEFORE mount

Reads localStorage.getItem('owismind.theme'), validates to 'dark' or 'light', falls back to 'light', and writes document.body.dataset.theme. Critical: the semantic tokens (--bg, --surface, --text...) only exist under body[data-theme]. Skipping this step produces a flash of unresolved tokens.

3. Create and configure the app

createApp(App).use(pinia).use(i18n).use(router) with pinia = createPinia().

4. DEV-only window hook (tree-shaken in production)

if (import.meta.env.DEV) window.__pinia = pinia. Exposes the store for manual seeding outside DSS.

5. Mount

app.mount('#app'). The #app target is declared in the source index.html.

HASH router

router/index.js uses createWebHashHistory(). This is an explicit, documented choice: DSS serves the webapp at a fixed URL with no server-side SPA rewriting. A path-based history (createWebHistory) would return a 404 on page reload or a deep link, because DSS cannot rewrite /chat/<id> to index.html. See ADR-0001. All views are lazy-imported to keep the initial chat bundle small.

i18n: pristine messages.json + extra.js

The i18n instance: createI18n({ legacy:false, globalInjection:true, locale:detectLocale(), fallbackLocale:'en', messages }). English is the default locale and the fallback. warnHtmlMessage:false allows a few trusted canned-HTML keys (such as default.answer_html).

Never edit messages.json by hand

messages.json is a 1:1 port of the original mockup's window.OWI_I18N and stays PRISTINE. Every new string or override goes into i18n/extra.js (flat key per locale: export const extraMessages = { fr: {...}, en: {...} }). The extra.js merge runs AFTER messages.json, so it can intentionally mask a key of messages.json.

Two domain catalogs are merged at setup via i18n.global.mergeLocaleMessage: timelineMessages from registries/timelineSteps.js (live timeline labels), and extraMessages from i18n/extra.js. Key families in extra.js: x.*, set.*, chat.*, ag.*, admin.* (including admin.quotas.*), ev.*, art.*, mode.*, ev.exp.* (trust layer steps). Positional placeholders use list interpolation: t('key', [arg0, arg1]).

Theming: Orange charter + body[data-theme]

src/styles/tokens.css is the single source of the theme layer. It provides two blocks: theme-independent tokens under :root, and semantic tokens under body[data-theme="light"] / body[data-theme="dark"] (the swap on body[data-theme] is the entire light/dark mechanism).

Orange charter bans (rule #10)

color-mix(), blur / backdrop-filter, gradients, glow / large soft shadows, emoji in UI, a global orange focus-ring, and any CSS-reconstructed brand mark. The brand logo is always the REAL image src/assets/orange-logo.png, imported as a module, never a colored CSS square.

The --orange-text token is a darkened orange (#b85700 in light mode) to pass WCAG AA 4.5:1 contrast on white. In dark mode it shifts to #ffb066. The status tints (--success-soft, --danger-soft) are more saturated in dark mode to avoid washing out on dark surfaces.

Theming gotcha (F2): a <style scoped> theme override must place the ENTIRE selector inside :global(body[data-theme="dark"] .x) - otherwise the scoped descendant combinator is lost.

State and stores

The frontend state is organized as a strict two-layer architecture: four REACTIVE Pinia stores hold application state, and around them orbit PURE modules (zero Vue imports, fully testable with node:test) that contain all deterministic logic.

Store and module map

The central reactivity model

This is the key mechanism behind all live streaming. Each chat exchange owns a "response version" object created as reactive(createAnswerState()) in chat.js. The pure reducer applyEvent (in composables/timelineModel.js) receives the stream of backend events and MUTATES this object in place: state.timeline.push(...), last.text += delta, state.status = 'done'.

The subtle point: applyEvent imports nothing from Vue (it is fully pure and testable outside the browser), but because the object it mutates is a reactive() proxy, each nested mutation triggers a fine-grained, live Vue re-render. The same pure function drives both the node:test tests and the real-time display.

The chat store

chat.js models the active conversation as a TREE of exchanges (not a flat list). Its key responsibilities:

MechanismDetail
Exchange shapereactive({ uid, id, parentId, userText, version, createdAt }). uid is the stable render key (never changes); id is null during a live run then reconciled.
Active pathturns = computed(() => buildActivePath(exchanges, overrides)). Pure read of the tree, delegated to conversationTree.js.
canSendRequires !sending && !threadLoading && !threadError && !session.needsConfig && !session.budgetBlocked && session.hasAgents && !!session.selectedAgentKey.
Cancellation tokenA closure { cancelled } checked before and after each await in the poll loop. Prevents stale events from corrupting a newer conversation.
Stop racestopPending covers the case where the user clicks Stop BEFORE /chat/start returned the run_id.

Three ways to create an exchange, illustrating the tree model:

ActionTree effect
send(text)Child of the last turn (conversation continuation)
editTurn(turn, newText)New sibling, same parent - nothing is deleted
regenerateTurn(turn)New sibling, same prompt (a new response version)
setTurnVersion(turn, idx)Pins a sibling via overrides; ignores a still-live sibling (id === null)

The session store

session.js carries four concerns: user identity, the list of activatable agents (feeding the picker and agents library), the paginated conversation list, and the monthly budget/usage status. The frontend never receives a raw agent_id: agents carries only an opaque logical key, a label, and admin-authored profile fields.

Key computed: budgetBlocked is true when usage.enforced !== false && usage.blocked, which disables canSend proactively. The server gate in /chat/start (402) is the real authority; this is a UI convenience.

The evidence store

evidence.js manages the Evidence Studio panel: the current exchange, the /evidence/meta response, editable chip state, and row pagination. Three sequence guards prevent stale responses from overwriting fresh state: seq for open/close, rowsSeq for out-of-order row responses, and userChipSeq for user-added chips.

Key invariant (rule F13): setActiveTab(key) NEVER touches open. The ChatThread auto-scroll gate observes evidence.open, not activeTab; changing the tab must never re-trigger the scroll.

The ui store: preferences

ui.js is the single source of truth for all user preferences. It replaces the window.STATE of the original mockup. The header (MainTop) and Settings page (SettingsView) both read and write it, so a change in one is reflected immediately in the other.

Chat family

ChatView.vue has three mutually exclusive branches:

1. session.needsConfig

Renders an administration card. The SQL connection is not yet configured by an admin.

2. chat.hasMessages

ChatThread + prompt area (prompt-wrap + PromptBar). Also renders the budget banner when session.budgetBlocked is true.

3. Otherwise (empty stage)

ChatEmpty + PromptBar inside prompt-wrap in-empty.

ChatThread.vue has critical rendering rules:

  • The v-for key is turn.exchange.uid (stable from creation), NEVER id (which is null during a live run and changes to the backend id when reconciled). Using id as key would cause a remount mid-stream.
  • Auto-scroll is gated on chat.sending: otherwise version navigation would recompute the signature and pull the view to the bottom. It NEVER watches chat.turns directly (every stream tick would re-trigger the scroll).
  • A user who has already scrolled up is never forcibly pulled down (NEAR_BOTTOM_PX = 120 threshold).

MessageAgent.vue renders an agent message in four zones: header, grouped activity block (steps/tool calls), response body (text/error blocks in arrival order), and the collapsible SQL panel + action footer. Its entire display model derives from pure selectors imported from timelineModel.js - it never mutates the state. While live, it renders in chronological segments with a ticker bounded to the last 5 lines (LIVE_WINDOW = 5). When terminal, all activity collapses to a single header line.

Backend communication

All backend traffic goes through services/backend.js. The frontend never hardcodes a URL: DSS injects window.getWebAppBackendUrl globally, and the helper backendUrl(path) resolves it lazily. If the resolver is absent (app launched outside the DSS webapp context), it throws immediately with a clear message.

The HTTP client

src/services/backend.js - request() JavaScript 
const res = await fetch(backendUrl(path), {
  credentials: 'same-origin',     // DSS auth cookies travel with every request
  ...opts,
  headers: { Accept: 'application/json', ...(opts.headers || {}) },
});
if (!res.ok) {
  let code = 'http_' + res.status;
  try { const data = await res.json(); if (data && data.error) code = data.error; } catch (e) {}
  throw new Error(code);          // e.message IS the stable error code
}
return res.json();

Three key design decisions: credentials: 'same-origin' (the DSS auth cookies identify the caller - no separate app token); stable error codes (on non-2xx, request() reads { error: "code" } from the JSON body and throws it; callers compare e.message); and Content-Type: application/json added only on POSTs (never on GETs).

The Stop flow

stopGeneration() posts /chat/stop, KEEPS polling (cooperative stop), and sets activeVersion.stopping = true (shows the "Stopping..." banner). The backend stop is cooperative: the LLM Mesh stream has no cancel API; the worker can only break between two chunks. The frontend waits for the terminal stopped event, which arrives after the backend emits final_answer then stopped. Every terminal event (including run_done and error) resets stopping = false.

Budget gate: monthly_quota_exceeded (402)

When /chat/start returns 402 (budget gate flipped between send click and actual start), the store DROPS the optimistic exchange from exchanges (no empty error bubble in the thread) and refreshes usage via session.loadUsage() so the budget banner appears immediately. No run was started, so no event will ever arrive.

Build and assets

The frontend is not served by a Node server. Vite compiles it once into hashed static bundles; those files travel inside the plugin and are served by DSS from the plugin resource folder.

From index.html to body.html

DSS serves the webapp (descriptor webapp.json, "baseType": "STANDARD") via body.html, not index.html. The final build step copies the built index.html to the webapp's body.html. Because the hash changes with every build, this copy MUST be done on every build - a stale body.html points to nonexistent assets (404).

The /build-plugin skill validates the copy by checking that the prefix /plugins/owismind/resource/owismind-app/ is present in body.html. Two companion files (app.js, style.css) are present alongside body.html but EMPTY of logic; DSS requires them and they must never be deleted.

Build output

npm run build writes into Plugin/owismind/resource/owismind-app/:

  • A rewritten index.html (all asset paths prefixed with base).
  • An assets/ folder with content-hashed bundles: the entry chunk assets/index-<hash>.js + CSS, lazy view chunks (ChatView-*.js, AdminView-*.js, etc.), shared chunks (Icon-*.js, session-*.js, budgetModel-*.js), and the bundled orange-logo-<hash>.png.
  • The favicon.svg copied from public/.

The logo is bundled, not path-referenced

Sidebar.vue imports the logo as import logoUrl from '../../assets/orange-logo.png'. Vite bundles it with a content hash. This is the ONLY correct way to include the logo: the bundled URL carries the correct base-prefixed path in production. A hardcoded path or a CSS-reconstructed placeholder violates the Orange charter (rule #10).
