OWIsMind

A business agentic chat portal, packaged as a Dataiku DSS plugin. Ask a question in natural language about Orange/OWI telecom revenues. Get a figure-based answer without writing a single line of SQL. Inspect the evidence behind every number: the exact SQL, the captured result, the scope and the cost. Not a demo. Not a consumer chatbot. A trusted, professional, data-oriented portal built on traceability.

The problem it solves

OWI/Orange business users need revenue figures, but those figures live in a SQL dataset that requires SQL skills to query. Two classic obstacles arise: you have to write SQL (a rare skill on the business side), and you have to trust an AI-generated answer (the risk of hallucination, of an invented figure, of a fuzzy scope). OWIsMind addresses both at once.

Productivity

No SQL required

Ask in French or English. The agent translates your question into a SQL query, executes it, formats the result in euros, and writes the analysis. No schema knowledge, no query syntax.

Trust

Evidence behind every number

Every figure comes from a real SQL result. The user sees the exact query, the captured rows, the scope (scenario, period, entity) and the cost. Nothing is generated from memory.

Extensibility

Configuration, not a rewrite

Adding an agent, an artifact type or a new domain is a matter of configuration: one registry entry, one Code Agent. The webapp contains no agent business logic.

The differentiating trio

The product is recognizable by three capabilities that work in concert. A single natural-language question triggers all three simultaneously.

The honesty firewall

Trust does not rest on a promise. It rests on a structural constraint. The orchestrator holds no business figure: every figure comes from a sub-agent that executed SQL. Therefore it structurally cannot invent a number. This is not a safety filter bolted on; it is the architecture.

Structural invariant

The orchestrator's four rules (from its system prompt)

  • It never produces an unsourced business fact, a figure, a source or a capability on its own.
  • It never tells the user that a measure, a scenario or a record is missing, null or unavailable. Only a specialist sub-agent can say that, after looking at the data.
  • The only allowed form of "no" is the capability gap: "there is no AGENT yet for this domain." It will never say "the data does not exist."
  • It never does arithmetic in its head. Exact sums, deltas, ratios and rankings are the specialist's work.

The trust badge: never green

On the Evidence Studio side, a product choice reinforces this stance. The verification badge is never green - to avoid any false assurance. Three levels:

Certified (solid)

Exact scope identified, result captured, SQL stored. Highest confidence - but shown in neutral, not green.

Partial (dotted)

Source identified or scope partially matched. Readable, but some details could not be confirmed deterministically.

Declared (gray)

Minimal evidence. The agent declared a result but the Evidence engine could not re-derive it fully from stored SQL.

Known limitations

These are deliberate architecture or V1 scope choices, not oversights.

No cross-dataset JOIN

The sub-agent works on one table, never a JOIN. An analysis spanning multiple datasets goes through the orchestrator (one agent per dataset). This requires a second staffed domain.

No SSE streaming - polling transport

DSS buffers SSE responses. The chosen solution is polling-via-thread: the frontend polls /chat/poll every ~500 ms. The text answer often arrives in one block at the end; the usable live view is the timeline.

Desktop-first, mobile out of scope

V1 targets desktop workstations from 12 inches to ultra-wide. Mobile is explicitly out of scope for this version.

Single-process assumption

The backend assumes a single process (in-memory _RUNS dict). Guardrails: max 8 concurrent runs, orphan TTL eviction, run queryable only by its owner.

Key terms at a glance

The full glossary is at 00-overview/03-glossary.md. These six terms appear on almost every page.

Term Definition
grounding Anchoring the user's typed terms onto exact cell values, via read-only inline SQL on the value index. Happens in the RESOLVE step. It is NOT a tool call.
Evidence Studio The right-hand panel that deterministically re-derives how a reply was produced (zero LLM call). Opens automatically when generation ends.
honesty firewall The orchestrator's structural rule: emit no unsourced business fact; the only allowed "no" is a capability gap, never "the data does not exist".
Phase / scenario The Phase column of DRIVE_Revenues: ACTUALS (default, always plural), BUDGET, FORECAST, Q3F, HLF. NEVER sum across phases.
capability gap The ONLY form of "no" the orchestrator may give: "there is no AGENT yet for this domain." Never "the data does not exist."
agent_key The opaque key (ag_<hash>) the frontend sends. The backend resolves it to (project_key, agent_id). A raw agent_id is never accepted from the front.
