What these records are for

An ADR (Architecture Decision Record) is a short document that captures a structuring choice: the context that made it necessary, what was decided, the reasons, the consequences (good and bad), and the alternatives that were ruled out. The goal is not to re-explain the code - the technical sections do that - but to answer the question every newcomer or future maintainer asks: "why is this done THIS WAY, and not otherwise?"

In OWIsMind, several choices look counter-intuitive at first glance: streaming goes through polling rather than SSE, the runtime uses no Dataiku Flow, the frontend routes in hash mode, and the agents run in a Python environment different from the backend. Each of these decisions has a concrete cause - often a failure observed in DSS - and a safety constraint behind it. The ADRs preserve this memory and prevent a future "simplification" from reintroducing a bug that has already been paid for.

Source of truth hierarchy

The ADRs are the hardened projection of the project's living memory (memory/LESSONS.md and memory/PROJECT_STATE.md). In case of divergence with an older guide in docs/cadrage/, the memory files take precedence.

Delivery and platform (ADR-0001, 0002)

Two foundational decisions set the delivery model. Both were forced by concrete constraints of the DSS hosting environment, discovered by testing, not by reading documentation.

ADR-0001

ADR-0001

Vue SPA served by DSS as static assets (hash router)

Accepted (DSS)

Decision: build Vue 3 + Vite into resource/owismind-app/, set base to the DSS plugin path, use createWebHashHistory(), and copy index.html to body.html after each build.

DSS serves the plugin's resource/ folder at an imposed fixed path and has no server-side URL rewriting for SPAs. A router in "history path" mode would return a 404 on page reload or deep-link sharing. Aligning Vite's base to the DSS serving path makes all asset links correct without post-processing. The hash keeps all routing client-side, so DSS always serves the same entry page. The NO INSTALL rule keeps library versions frozen: only the user installs, never the agent.

  • Simple deployment: a static-asset zip upload, no server runtime on the frontend side.
  • Robust reload and deep links: #/chat/<sessionId> reopens without a 404.
  • Hash-based URLs (#/route) are less clean than pure paths - an accepted trade-off of the DSS model.
  • body.html must be rewired after every build; never edit it by hand.

ADR-0002

ADR-0002

Streaming by polling (no SSE, no WebSockets)

Accepted (DSS)

Decision: drop SSE; use polling-through-a-thread (POST /chat/start spawns a daemon thread, GET /chat/poll?run_id=&cursor= returns new events since the cursor, polled every ~500 ms until done).

SSE was the first implementation. On the real instance it failed: the DSS nginx proxy buffered the entire long response and delivered all events at once at the end. The X-Accel-Buffering: no header is not guaranteed to be honored from a standard webapp backend. The fix was copied from the client's production Dash app running on the SAME instance - it never exposes a long HTTP response, so it bypasses buffering by design. Each /chat/poll response is a short, fast request (3-4 ms measured in DSS), immune to buffering. The cursor keeps only new events per poll, so cost per call stays constant.

Safety guardrails added: MAX_CONCURRENT_RUNS = 8, FINISHED_TTL_SECONDS = 60, HARD_TTL_SECONDS = 600, MAX_RUN_SECONDS = 300, owner-scoped run ids, per-user anti-spam gate MIN_START_INTERVAL_SECONDS = 1, and per-run memory bounds.

  • Transport proven in DSS. Zero buffering at the proxy. Multi-agent flow works.
  • No long HTTP response, so no proxy buffering surface. Run is bounded and safe.
  • No word-by-word typing. The live content that is usable is the timeline of steps, not letter-by-letter prose.
  • A backend restart loses in-flight runs (_RUNS is in-process memory); handled gracefully by the frontend terminal code.

Safety and storage (ADR-0003, 0004)

These two decisions define the trust boundary and the data safety posture. Both are NON-NEGOTIABLE project rules (rules 3 and 4 in CLAUDE.md), audited adversarially and confirmed in DSS.

ADR-0003

ADR-0003

Direct SQL, no Flow at runtime

Accepted (DSS)

Decision: all runtime reads and writes go through SQLExecutor2 with parameterized queries, explicit COMMIT, a fresh executor per call, and read-only transactions on replay paths. No DSS Flow at runtime.

OWIsMind runs on a shared Dataiku instance. Flow execution touches the scheduler and orchestration of the entire instance - unsuitable for KB-sized transactional writes on every chat request. Direct SQL is predictable, auditable, and bounded. Key invariants: fresh executor per call (avoids shared transaction state across Flask worker threads), parameterized values via sql_value() (never string-concatenated), systematic COMMIT in post_queries, read-only transactions on Evidence replay paths, and versioned _vN tables instead of destructive ALTER. Schema naming: public."PROJECT_KEY_owismind_tablename".

Non-negotiable rule

There is NO generic SQL route. The frontend never chooses the table, connection, or query. No /execute-sql endpoint exists. This is rule #3 in CLAUDE.md.

  • No injection surface: adversarial greps for DROP/TRUNCATE/set_* came back empty.
  • 6-lens Evidence audit: 0 injection / 0 IDOR / 0 XSS.
  • Schema changes require a new _vN table (old conversations become invisible at switchover - accepted).
  • SQLExecutor2 always inlines values into query text (logged by DSS); hence the persisted-text cap.

ADR-0004

ADR-0004

Server-side agent whitelist (opaque logical key)

Accepted (DSS)

Decision: the frontend only ever sees an opaque logical key (ag_<sha1[:12]>). The backend is the single point of resolution to (project_key, agent_id) via an administered whitelist in webapp_settings_v1. A raw agent_id never crosses the trust boundary.

The frontend is an untrusted client. Giving it a raw agent_id (e.g. agent:bHrWLyOL) would allow any user to run an agent that the admin never authorized. The opaque key is derived deterministically from sha1(project_key:agent_id)[:12] so that re-saving the same selection yields the same key (conversations stay attached). Admin saves re-validate every agent against the live DSS listing: a forged or invisible project id can never be persisted. GET /agents returns only {key, label}; POST /chat/start enforces the whitelist via resolve_enabled_agent().

  • Validated in DSS. No table readable by the frontend exposes an agent_id or project_key.
  • Adding or removing an agent is a purely administrative operation - no code change, no schema migration.
  • If an agent is deactivated or moved to another project, its logical key stops resolving and old conversations referencing it become orphaned.

Agent intelligence (ADR-0005 to 0011)

Seven decisions shape the agent layer. Together they define the runtime (LangGraph on Python 3.11), the LLM integration discipline (native Mesh calls, forced JSON only on UNDERSTAND), the data-trust contract (Evidence Studio), the model-agnostic mode system, the data engine (Semantic Model + grounding), and the sub-agent assistance principle.

ADR-0005

ADR-0005

LangGraph Code Agents in Python 3.11 (3.9/3.11 dual path)

Accepted (DSS)

Decision: both agents are LangGraph StateGraph programs in self-contained Python files, running on a Python 3.11 code env, distinct from the Flask backend (Python 3.9.23). The repository is the source of truth; agents are re-pasted by hand into DSS after every change.

LangChain and LangGraph require Python 3.10 or higher; the Flask backend runs on 3.9.23 (observed, not assumed). Both code envs exist on the instance simultaneously - the dual path is the platform reality. DSS visual agents are black boxes that give no control over the loop, frozen event contracts, deterministic evidence capture, or mode propagation. LangGraph brings an explicit typed StateGraph loop and a custom stream writer for the timeline, with full code control. get_stream_writer() works in synchronous nodes on Python 3.11 (proven in the DSS log, lesson L055) - a key reason to stay on 3.11 and keep nodes synchronous.

  • Validated in DSS: the DSS log proves LangGraph runs on the 3.11 Code Agent.
  • Full control of the loop, contracts, evidence capture, and mode propagation.
  • Manual re-pasting: every repository change requires re-pasting BOTH Code Agents into DSS (env 3.11).
  • Two separate processes: config set in one does not reach the other without being explicitly routed.

ADR-0006

ADR-0006

Native LLM Mesh calls in nodes (no as_langchain_chat_model)

Accepted (DSS)

Decision: all LLM calls inside nodes use native Mesh APIs - project.get_llm(id).new_completion() then execute() / execute_streamed(), and project.get_agent_tool(id).run() for tools. Never as_langchain_chat_model().

The as_langchain_chat_model() adapter normalizes the exchange to the LangChain format and loses the Mesh's native reasoning and tool-calling. Without reasoning, the orchestrator cannot route reliably; without tool-calling, it cannot delegate to sub-agents or request chart rendering. Native calls also preserve the exact tool_call -> tool_output pairing required by Claude/Vertex (a broken pairing triggers a 400). Additionally, synchronous nodes with native blocking Mesh calls work correctly with get_stream_writer() on Python 3.11 - the async path has a known bug for this.

  • Reasoning and tool-calling preserved end to end. Evidence capture is deterministic (SQL and rows read from the tool return value, not guessed from a trace).
  • The tool_call / tool_output pairing is under direct code control - avoids 400s from the Mesh API.
  • The agent files must be self-contained (only stdlib, dataiku, langgraph; no plugin imports). Re-pasting is required after every change.

ADR-0007

ADR-0007

with_json_output forced on UNDERSTAND, never on the orchestrator

Accepted (DSS)

Decision: the sub-agent's UNDERSTAND step calls completion.with_json_output(schema=...) for a clean, fast parse. The orchestrator NEVER calls with_json_output because in DSS 14 it silently disables the model's reasoning.

Without forced JSON, the UNDERSTAND step's model "thought" for 15 seconds then returned prose the parser could not read. The sub-agent would short-circuit the entire pipeline with an internal error before a single SQL query ran (lesson L056). The general rule: with_json_output for any output consumed by code (deterministic extraction), and reasoning reserved for decisions and prose (the orchestrator's routing and the verified headline). In DSS 14, native JSON mode disables reasoning for that call - which is exactly what we want for UNDERSTAND (an extraction does not need to think) and exactly what we must avoid for the orchestrator (which must reason to route correctly).

  • Validated in DSS: extraction is reliable and fast; the internal-error-before-SQL bug is gone.
  • Schema anchored on the profile enums: the model cannot produce an out-of-domain value.
  • If JSON mode becomes unavailable on the connection, the prompt-only fallback (attempt 2) is less reliable on small models. Traced and observable via span attributes.

ADR-0008

ADR-0008

Evidence trust layer and artifacts (signal vs data separation)

Accepted (DSS)

Decision: Evidence Studio re-derives all proof from the stored SQL on every call, using ZERO LLM in the path. Artifacts carry only a display SPEC (x, y, chart_type); the data rows come from the already-captured result, never from the agent.

OWIsMind targets non-technical sales people who must be able to trust a revenue figure. Putting an LLM in the evidence path would make the evidence itself a generation - and false evidence is worse than no evidence. The trust level is a deterministic scale computed by pure static analysis of the stored SQL (levels: declared to calc_decomposed); the badge is NEVER green. For artifacts: binding the Chart.js payload to the single captured SQL result guarantees that the chart is by construction consistent with the Evidence tab. No duplication of sensitive data in a dedicated schema; everything is re-derived from generated_sql already stored in the chat row.

In Flux: result capture is best-effort

The exact key of the rows in the tool span is not confirmed on the instance. Capture probes several candidate keys (_ROW_KEYS); if absent, the panel shows result_captured: false without guessing. The trust level remains valid independently (it only depends on the stored SQL text).

  • Evidence can never over-assert: every degradation path lowers the level, never raises it.
  • Chart and evidence are always consistent (same captured result rows).
  • Without an admin dataset whitelist (MULTISELECT does not render in DSS Settings), any SQL dataset in the project whose table the agent queries is visible as raw rows in Evidence.

ADR-0009

ADR-0009

Per-mode model, no escalation (model-agnostic architecture)

Accepted (in flux)

Decision: one model drives the entire turn, chosen by the user mode (eco = Gemini 3.1 Flash-Lite, medium = Gemini 3.5 Flash, high = Claude Sonnet 4.6). No escalation. The mode propagates to the sub-agent via a machine token ⟦owi:mode=...⟧ appended at the END of the message by the backend.

The previous escalation architecture failed in DSS on gpt-5.4-mini: either the model escalated systematically (triggering a hardcoded transition message) or it narrated its intent then stopped without calling a single tool ("narrate-and-stop"). User instruction: stop the single-model hacks, aim for an architecture that works even on small models. A single model per mode is predictable and genuinely model-agnostic. Narration is OFF in eco mode (removes the "speak before acting" prompt that caused narrate-and-stop on the smallest tier). The token is read LAST by the orchestrator so that a user-typed fake mode token earlier in the message cannot force a more expensive model.

In Flux: Gemini model ids must be verified

The constants GEMINI_FLASH_LITE_ID and GEMINI_FLASH_ID follow the format observed on the LLM-7064-revforecast connection but must match the target instance exactly. A wrong id breaks the corresponding mode. Eco is the DEFAULT - a wrong eco id means no response at all.

  • Predictable: orchestration logic is identical across all three modes, only the tier changes.
  • The Semantic Model Query tool runs on Sonnet in ALL modes - SQL quality stays constant.
  • Both Code Agents carry the same LLM_BY_MODE table. An id change must be reflected in BOTH files, then both re-pasted in env 3.11.

ADR-0010

ADR-0010

Grounding via value_index; Semantic Model owns the SQL (hybrid engine)

Accepted (in flux)

Decision: the default engine is SQL_ENGINE = "semantic_tool". The sub-agent runs UNDERSTAND/RESOLVE/COMPOSE then delegates SQL generation AND execution to the DSS Semantic Model Query tool (v4oqA6R). Term grounding is inline read-only SQL on the DRIVE_Revenues_value_index dataset (NOT a separate DSS tool call), via a three-stage cascade: exact match, fuzzy ILIKE, last-chance difflib scan.

A user A/B test in DSS showed that the Semantic Model (Sonnet + semantic layer) "answers and understands much better" than our hand-written SQL templates. SOTA research confirms it: semantic-layer-guided SQL achieves 98-100% accuracy vs 84-90% for free LLM-generated SQL. The code's job is to give the model the best possible context. Grounding stays inline SQL (not a tool) because a tool would run the same SQL with no gain; keeping it inline means bounded, read-only, database-side matching with no extra tool surface. The direct SQL engine is retained as a technical fallback (FALLBACK_TO_DIRECT = True) but is never used for a legitimate empty result.

In Flux: attribute_lookup wiring

The attribute_lookup tool (tools/attribute_lookup_tool.py) replaces the former dataset_lookup managed tool (removed 2026-06-18). It is wired as a built-in of the ORCHESTRATOR (LOOKUP_TOOL_ID, empty until the Custom Python tool is created in DSS). The sub-agent stays unchanged. Confirmation on the instance is pending.

Roadmap

DRIVE_Revenues_Value_Catalog and the Python resolver Drive_Revenues_resolve_filter_value are NOT wired in v3. They are superseded by attribute_lookup for the direct-lookup need.

  • Validated in DSS: the data engine works ("really well"). Grounding is safe and bounded with no hardcoded business value.
  • Clean separation: code owns UNDERSTAND/RESOLVE/COMPOSE and instance safety; Sonnet owns SQL generation and the business hierarchy.
  • The Semantic Model engine is opaque: anomalies are fixed in its config, not in the agent code. Alignment scripts: tools/semantic_model/build_aligned_semantic_model.py.
  • The value_index can be stale after a dataset change; freshness depends on running the Flow recipe.

ADR-0011

ADR-0011

Assistive sub-agent (does not pin a column for an ambiguous offer term)

Accepted (in flux)

Decision: when a user term appears in multiple offer hierarchy columns, the sub-agent emits an AMBIGUOUS OFFER TERM marker in the message to the Semantic Model and lets the model decide the hierarchy. It never pins the column itself. Single-column ambiguities (two distinct entities in one column) still trigger a clarification request.

A concrete bug triggered this decision: "revenus YTD EVPL, actuals vs budget". The earlier sub-agent used column_priority (-distinct_count fallback) which ranked sirano_product (153 distinct values) over Product (42 distinct values) and pinned sirano_product = 'EVPL'. Budget rows carry no sirano_product, so the budget result was 0. The Semantic Model (Sonnet, Playground) resolved Product = 'EVPL' perfectly. The small sub-agent was dictating the wrong answer to a model that would have known better. The business hierarchy (Product > Solution > SolutionLine > sirano_product, "never default to sirano_product") lives in the model's instructions, not in the code (project rule P3: no business value hardcoded in agent logic).

In Flux: EVPL smoke test pending

The EVPL case must be re-tested via the orchestrator in DSS: "revenus YTD EVPL, actuals vs budget" should resolve to Product = 'EVPL', not sirano_product, and the budget result should not be zero. Validation pending.

  • Fixes the budget = 0 trap on multi-column offer terms.
  • P3-clean: zero hardcoded column name or hierarchy in the agent code.
  • The guarantee depends on the Semantic Model instructions. If "never default to sirano_product" is removed from the model config, the bug can return on the model side.

Product identity (ADR-0012 to 0015)

Four decisions govern the non-functional identity of the product: the typographic rule that eliminates an AI signature from every surface, the admin-authored agent profiles system, the monthly budget enforcement layer, and the sober Orange design charter. The three decisions added in the 2026-06-18/19 sessions (ADR-0013, 0014, 0015) are the most recent in the repository.

ADR-0012

ADR-0012

Typographic rule: no em dash or en dash, anywhere, ever

Accepted

Decision: U+2014 (em dash) and U+2013 (en dash) are banned on every surface of the project: i18n strings, UI text, source code, comments, project memory, commit messages, and chat responses. Use -, :, , or parentheses instead.

Language models produce em dashes abundantly for asides and pauses; their presence in a project where AI assists the authoring betrays that origin and harms the project's identity. User decision of 2026-06-17, no functional exception. The verification is deterministic and inexpensive: grep -rlP '\xe2\x80\x9[34]' . (byte-safe, covers both glyphs). The sweep must be byte-safe using LC_ALL=C to avoid corrupting multibyte agent control tokens (⟦owi:mode⟧, logical symbols). A pre-commit git hook was proposed but is not yet installed; manual grep is the current barrier.

Shell 
# Verify: zero em/en dashes anywhere in the repo
grep -rlP '\xe2\x80\x9[34]' .

# Verify: no U+FFFD corruption from a previous sweep
grep -rlP '\xef\xbf\xbd' .
  • Homogeneous identity: the whole corpus (code, doc, memory, commits, chat) speaks the same typography.
  • Verifiable in one command, integrable into any pre-commit check or review.
  • No automatic guardrail yet (pre-commit hook not installed); discipline and manual grep act as the safety net.

ADR-0013 - new (2026-06-18)

ADR-0013 New

Agent profiles are admin-authored (no hardcoded copy)

Accepted

Decision: agent display copy (tagline, description, capabilities, tools, icon, badge) is authored by an administrator through the Admin UI, validated by validate_agent_meta (pure, never raises, clamps to bounds), stored inside the enabled_agents JSON in webapp_settings_v1, and exposed via GET /agents without ever leaking a raw agent id.

The previous agentMeta.js frontend module hardcoded descriptions that went stale whenever an agent was renamed or restructured, and required a full frontend rebuild + zip upload for any text update. Deriving copy from the DSS agent's own technical label ("OWIsMind_orchestrator") is correct for raw admin views but inadequate as a user-facing card. The server-side constraint: the frontend must never receive a raw agent_id (ADR-0004), so the profile must be keyed by the opaque logical_key. Piggybacking on the existing webapp_settings_v1 avoids a new table, stays within the no-ALTER discipline, and keeps the enabled-agents lifecycle in one place. validate_agent_meta never raises so a malformed admin payload degrades gracefully without blocking the whitelist save.

  • Live copy editable by an admin with no code change, no rebuild, no deploy.
  • An empty profile renders an honest "profile to complete" placeholder, never stale hardcoded text.
  • The admin must fill in each profile before the library feels complete to users. Until then, the placeholder card is shown.
  • Adding a new icon requires updating both ALLOWED_AGENT_ICONS in validation.py AND the frontend icon registry - one backend restart required.

ADR-0014 - new (2026-06-18)

ADR-0014 New

Monthly rolling budget and per-user quota overrides

Accepted (in flux)

Decision: a rolling 50 USD/month credit per user, enforced at POST /chat/start (HTTP 402, fail-open on storage error). The limit resolves via a two-layer cascade: per-user override in webapp_user_quota_v1 (permanent or time-boxed) over a global default in webapp_settings_v1. Monthly spend accumulates in webapp_usage_monthly_v1 as calendar-bucket rows - no reset cron job needed.

Without a spending ceiling, a single user could exhaust the organization's LLM Mesh quota calling Sonnet in high mode indefinitely. The calendar-bucket approach (primary key = (user_id, period_start), where period_start = date_trunc('month', now())) means a new month is simply a new row; no scheduled job can be missed. The fail-open contract - allowing the request through on a storage error - prioritizes chat availability over perfectly-timed enforcement; the spend is still recorded, so the next request is correctly gated. The 30-second in-process config cache removes one DB round-trip per /chat/start call. A new _v1 table holds only the per-user exceptions (most users use the global default), preserving the no-ALTER discipline.

In Flux: backend restart required

Not yet validated in DSS. A backend restart is required after uploading the plugin zip (python-lib changed). The webapp_user_quota_v1 table is created lazily on the first budget-admin write - no migration required upfront.

  • Rolling monthly reset requires no cron job: new month = new bucket row.
  • Budget card transparent to the user: shows spent, limit, source of the limit, and reset date.
  • Fail-open means a user with an exhausted budget still gets an answer during a DB outage. This is the intended trade-off.
  • Admin config changes propagate with up to 30s delay to other backend processes (if multi-process mode).

ADR-0015 - new (2026-06-18/19)

ADR-0015 New

The Orange charter: sober design system for the OWIsMind UI

Accepted (in flux)

Decision: white and near-black carry the layout; a SINGLE orange (#FF7900) is used as a RARE accent only on active states, eyebrows, title-bars, KPI rules, and primary actions. Square geometry everywhere (border-radius: 0; only avatars are round). Flat fills, 1px borders, heavy headings. All colors through semantic tokens (frontend/src/styles/tokens.css). The brand logo is ALWAYS the real orange-logo.png image, never a CSS-generated substitute.

Iterative UI work had accumulated rounded cards, glow effects, backdrop-filter blur, color-mix() that broke dark mode, and at one point a logo reconstructed in CSS. An Orange enterprise product must feel like the Orange brand: clean, authoritative, editorial, not decorated. The Orange charter formalizes this: a self-contained reference (docs/cadrage/CHARTE_ORANGE_UI.md) that replaces the original HTML mockup (now deleted). Key ban reasons: color-mix() produces invisible text in dark mode; blur is GPU-heavy on low-power devices; a CSS-generated logo scales poorly and requires maintenance; a global focus ring on the input container looks like an interactive brand element.

The refonte delivering this charter was built by 6 parallel Sonnet agents (2026-06-19) and is packaged as build index-BHeG2NRY.js (79 entries). Validated locally. Pending: upload the zip (frontend only, no backend restart unless the python-lib changes from the same session are not yet live); smoke-test the real logo in rail and sidebar, square geometry on cards and modals, light/dark theme.

  • Dark mode robust by construction: token-driven, no per-component theme override needed for standard cases.
  • Consistent, predictable UI that survives future contributions: bans are checkable by grep.
  • Square geometry is a hard constraint: any external template or snippet must be un-rounded before use.
  • The charter MUST be read before any styling task. Three incidents in the project history show what happens when it is skipped: rounded modals, a CSS-generated logo, a global focus ring.

Adding a new ADR

Any future structuring decision deserves an ADR. A decision is structuring if it is durable, costly to reverse, or surprising (it runs counter to a reasonable expectation). Typically concerned are: a change of transport, a new _vN table, a new agent or DSS tool, a change of safety posture, a new LLM model or mode, or an evolution of the Evidence proof contract.

Procedure

  1. Create 08-decisions/00NN-short-title-in-kebab-case.md with the next available number.
  2. Follow the template: Status, Context, Decision, Reasons, Consequences +/-, Rejected alternatives, See also.
  3. Do NOT duplicate a major diagram: an ADR illustrates a DECISION and points to the diagram in the relevant technical section.
  4. Add the corresponding line to the ADR table in 08-decisions/README.md.
  5. If the decision corrects an earlier one, mark the old ADR "Superseded by ADR-NN" instead of deleting it.

ADR golden rule

An ADR records WHY, not a how-to. The how lives in the 02-architecture, 04-backend, and 05-agents sections. An ADR that starts copying code or duplicating an operations guide has left its role.
