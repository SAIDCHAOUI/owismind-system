Installation and configuration

OWIsMind is deployed as three independent building blocks: a DSS plugin zip (Flask backend + built Vue frontend), two LangGraph Code Agents pasted by hand (outside the zip), and a small runtime configuration in the webapp Settings. This section walks through the complete sequence, from the first upload to the first conversation.

Two code environments: intentional

The Flask backend runs on Python 3.9 (no langchain). The two Code Agents need Python 3.11 (LangGraph). These two envs are separate by design - you cannot put langgraph in the 3.9 backend. The agents never enter the zip; they are deployed by copy-paste onto the 3.11 env. See ADR-0005.

Install the plugin

The plugin is distributed as owismind-upload.zip (canonical content: plugin.json at root + python-lib/ + resource/ + webapps/). Upload is a manual operator action.

  1. In DSS, open Plugins (plugin administration).
  2. Click Add plugin > Upload, choose owismind-upload.zip, set Origin = Uploaded.
  3. Let DSS install (plugin id owismind).

Development plugin trap

A Development plugin carrying the same id owismind cannot be updated by a ZIP upload. To keep the same id (and therefore the Vite asset paths already wired in body.html): delete the existing Development plugin first, then upload the ZIP with Origin = Uploaded.

Webapp settings

Once the plugin is installed, create a webapp from the webapp-owismind-ai-agents component and attach the Python 3.9 code env. Then configure four parameters in the webapp's Settings tab.

ParameterTypeRequiredRole
sql_connection Dynamic SELECT Yes (app does not function without it) The PostgreSQL connection where the backend stores its tables. Only PostgreSQL connections are listed.
table_prefix STRING No Optional prefix (max 16 chars, [A-Za-z0-9_-]) - allows several OWIsMind instances on the same connection. An invalid prefix is silently ignored.
traces_dataset Dynamic SELECT No SQL-table-backed dataset (3 columns: exchange_id, trace, created_at) for offline trace analysis. (none) disables it. Best-effort write - a failure never breaks the conversation.
log_level SELECT No (default INFO) Verbosity of the webapp backend log. Choices: DEBUG / INFO / WARNING. Requires backend restart to apply.

Storage is lazy

Tables (webapp_chat_v5, webapp_users_v1, webapp_settings_v1, webapp_usage_monthly_v1, webapp_user_quota_v1, webapp_artifacts_v1) are created on first use via CREATE TABLE IF NOT EXISTS. No manual DDL is needed once the connection is chosen. MULTISELECT-type params do NOT render in DSS Settings - none of OWIsMind's parameters use that type.

First admin, agent whitelist, and budget

First admin (TOFU): the very first user who POSTs /owismind-api/me after storage is configured becomes the application admin. The election is serialized by a PostgreSQL advisory lock. Ensure it is the deploying admin who opens the app first.

Agent whitelist: pasting Code Agents is not enough. An admin must explicitly activate each agent via the Administration page. The frontend only sends an opaque logical key (ag_<hash>); the backend resolves (project_key, agent_id) only if the agent is enabled. Activate at minimum the orchestrator OWIsMind_orchestrator.

Budget: the default credit is $50 USD per calendar month per user (DEFAULT_MONTHLY_LIMIT_USD = 50.0 in storage/budget.py). It is live immediately after deploy. Adjust via the Quotas tab in the admin space. The gate returns HTTP 402 (monthly_quota_exceeded) when a user's limit is reached; enforcement can be disabled globally.

Agent profiles

Agent descriptions are never hardcoded (the legacy agentMeta.js was removed). An admin authors each agent's library card via Administration > Agents > Edit profile. Until filled, the card shows a "profile to complete" placeholder. The profile is validated and sanitized server-side (validate_agent_meta in security/validation.py, pure, never raises, icon whitelist-checked) and stored in the enabled_agents JSON of webapp_settings_v1 - no new table. No restart is needed after saving.

Installation checklist

Complete sequence - zero to first conversation

  1. Build a Python 3.9 code env (backend) and a Python 3.11 code env (agents, with langchain/langgraph installed by the operator).
  2. Ensure a PostgreSQL DSS connection exists (by convention SQL_owi) and the backend run-as-user can write to it.
  3. Upload owismind-upload.zip with Origin = Uploaded (delete any Development plugin of the same id first).
  4. Create the webapp from webapp-owismind-ai-agents, attach the 3.9 code env.
  5. In Settings: select sql_connection; optionally fill table_prefix, traces_dataset, log_level.
  6. Start/Restart the backend, then force-refresh the browser.
  7. Paste the 2 Code Agents (OWIsMind_orchestrator + SalesDrive_revenue_expert) on the 3.11 env; verify all LLM Mesh ids.
  8. Open the app as the deploying admin (POST /me -> TOFU election).
  9. Via the admin space, activate at minimum the orchestrator OWIsMind_orchestrator.
  10. Review or adjust the default $50 USD/month budget via the Quotas tab.
  11. In Administration > Agents, click Edit profile on each enabled agent and fill in the library card.
  12. Smoke test: GET /owismind-api/ping must return {"status":"ok","python":"3.9.23"}; /admin/storage must show configured: true.

The plugin lifecycle: build, package, deploy

This is the most operationally critical section. The Vue/Vite frontend compiles into static assets, the Flask backend and the webapp ship in a plugin zip, and the two LangGraph Code Agents are pasted by hand (Python 3.11 env). Three independent flows, one cardinal rule:

Cardinal rule

Building does not package. Packaging does not upload. The agent never uploads. Uploading the zip and pasting the Code Agents are manual operations performed by the operator.

The build-to-zip pipeline

Step 1: /build-plugin (compile the frontend)

The /build-plugin skill chains five steps and never installs anything:

  1. Preflight: verify Plugin/owismind/frontend/node_modules/ exists. If missing, STOP - only the user installs (NO INSTALL policy).
  2. Build: npm --prefix Plugin/owismind/frontend run build (= vite build).
  3. Rewire body.html: copy the built index.html to webapps/webapp-owismind-ai-agents/body.html.
  4. Verify: confirm the asset base /plugins/owismind/resource/owismind-app/ is present in body.html.
  5. Report: remind the operator that packaging is a separate step.

The canonical Vite config (Plugin/owismind/frontend/vite.config.js) sets three values that must never change without a full rebuild:

Vite keyValueRole
base/plugins/owismind/resource/owismind-app/URL prefix of assets served by DSS
build.outDir../resource/owismind-appOutput (relative to frontend/)
build.emptyOutDirtruePurges the output folder before writing

body.html must be rewired on EVERY build

The bundle filename is HASHED and changes on every build. Without the copy of index.html to body.html, DSS serves a body.html pointing to an old hash and assets return 404. The cp may be refused by harness permissions - fallback is to rewrite body.html via the Write tool (gotcha F10). NEVER build into resource/ outside the skill: emptyOutDir: true would overwrite the deployed app.

For a compile-check only (without touching the deployed app):

Shell 
# Throwaway compile-check - NEVER build to resource/ outside /build-plugin
./node_modules/.bin/vite build --outDir /tmp/owi_buildcheck --emptyOutDir
rm -rf /tmp/owi_buildcheck

Step 2: /package-plugin (stage the runtime and zip)

The /package-plugin skill stages the runtime ONLY. Precondition: the frontend must already be built and body.html wired. The skill never uploads.

  1. Reset the staging folder (ready-for-dataiku/owismind-upload/ and zip).
  2. Stage the runtime: plugin.json at the root, then python-lib/, resource/, webapps/ copied via cp -R.
  3. Zip from the staging folder so that plugin.json is at the root.
  4. Verify the archive is clean and contains the required files.

Exclusion glob trap (lesson L002)

Zip exclusions are done BY NAME only (CLAUDE.md, README.md, __pycache__/, *.pyc). A broad glob like *.py or *.md would sweep up the python-lib/owismind/**/__init__.py files and break the from owismind.api.routes import register_routes import at runtime.

Why the built frontend is versioned in git

The built payload (resource/owismind-app/) is tracked in git - a deliberate exception to the "regenerable outputs = ignored" philosophy. The NO INSTALL policy means a fresh clone cannot reinstall npm and rebuild. So the payload must travel inside the repository to stay packageable from any clone. Never edit it by hand; never commit it outside of a /build-plugin run.

Decision matrix: what to rebuild, restart, re-paste

This is the table to remember for every type of change.

Frontend-only change

Build + package + upload + browser refresh. No backend restart needed.

Backend/python-lib change

Package + upload + RESTART backend (mandatory - old code runs until restart).

Agent-only change

Re-paste BOTH Code Agents (3.11 env). No zip, no upload, no restart.

Monitoring and logs

OWIsMind does not ship a dedicated observability stack. Monitoring relies on two sources: the DSS webapp backend log (primary, real-time) and an optional traces dataset (offline analysis, write-only).

The webapp backend log

The Flask backend writes all logs to the DSS webapp backend log for the webapp-owismind-ai-agents webapp (accessible via the Log / View backend log button in DSS). Each module logs as owismind.<module>. Two structuring lines appear at startup:

  • OWIsMind storage status: {...} - the full storage_status() dictionary (connection, project key, namespace, table names). This is the first thing to check after any deploy.
  • OWIsMind API ready - N routes: [...] - the live route table. If an expected route is missing, the wrong build is running.

Two blueprint-scoped hooks trace every /owismind-api/* request:

Log format 
→ GET /owismind-api/agents              # request starts
← GET /owismind-api/agents -> 200 (14 ms)  # request ends with duration

Content-free by design

The content of user messages and agent responses is never logged. Only metadata (length, identifiers, statuses) appears. This allows sharing log extracts without exposing business conversation content.

  • /chat/start logs: /chat/start - user_id=... session_id=... agent_key=... msg_len=... - message length only, never the text.
  • Authentication header values are never logged. Only the names of keys present are logged on identity failure.
  • When a run fails, internal details stay in the backend log server-side; the client only receives a stable error code.

Identifiers for correlation

IdentifierScopeWhere it appears
user_idDSS login of the caller (from browser auth headers)/chat/start, run start line, Evidence observability
session_idOne conversation (groups multiple exchanges)/chat/start
exchange_idOne user turn + response, one DB row (uuid4().hex)/chat/start return, run end lines, Evidence observability
run_idIn-flight generation cycle (ephemeral, in-memory only)All stream_manager - ... lines

Evidence observability

Each call to /evidence/meta writes a structured observability line:

Log format 
/evidence/meta - user_id=... exchange_id=... available=... reason=... level=... result_captured=... drill_available=... artifacts=N

result_captured=false is a normal possible state (best-effort capture), not necessarily an incident.

Run concurrency caps

GuardrailValueHTTP codeLog line
Global concurrent runsMAX_CONCURRENT_RUNS = 8503 busyconcurrency cap reached, rejected
Per-user spacingMIN_START_INTERVAL_SECONDS = 1.0 s429 rate_limitedrejected before write: rate_limited
Wall-clock deadlineMAX_RUN_SECONDS = 300.0 sterminal eventended ... early (...)
Abandon (tab closed)ABANDON_AFTER_SECONDS = 30.0 sterminal eventended ... early (...)

Single-process assumption

All state (_RUNS, identity cache, rate-limit buckets) is per-process. The backend MUST run as a single process. In multi-process mode, the concurrency cap would be multiplied by N and a cross-process /chat/poll would return 404. Verify at deployment.

Runbooks

Nine incident procedures, each following the same structure: symptom, checks, resolution, checklist. The most important cross-cutting reminders:

  • Transport = polling by thread (no SSE). The frontend polls /chat/poll every 500 ms. The text answer often arrives in one block at the end; the live signal is the timeline, not the prose.
  • The Flask backend (Python 3.9) and the two Code Agents (Python 3.11) are two distinct processes. A change in python-lib/ requires Restart backend; an agent change requires re-pasting the two Code Agents.

Runbook 1: agent no longer answers

Symptom: timeline shows "Parsing the request" and does not advance, or the spinner keeps turning without a final answer.

#CheckInterpretation
1Mode High?A long Sonnet call is normal. Wait at least 20-30 s before concluding failure.
2Backend restarted recently?A restart kills in-memory state _RUNS; the in-flight run is lost.
3Browser console: run_not_found?Terminal code - the run has disappeared. User can simply resend the question.
4Backend log: agent_unavailable?The agent raised, or the deadline MAX_RUN_SECONDS (300 s) cut it off.
5All runs blocked?Check GET /owismind-api/ping; if backend does not answer, do Restart backend.

Runbook 2: "storage not configured" (HTTP 409)

Symptom: the app refuses to send messages or list conversations; backend returns storage_not_configured (HTTP 409).

Cause: no SQL connection is selected in webapp Settings. The backend never guesses a connection.

Resolution: open webapp Settings, select sql_connection (preferably SQL_owi), restart the backend, force-refresh the browser.

Runbook 3: "agent not enabled" (HTTP 404)

Symptom: backend returns agent_not_enabled (HTTP 404); backend log shows /chat/start - agent_key not enabled: <key>.

Cause: the opaque logical key sent by the frontend does not resolve to an enabled agent in the server whitelist.

Resolution: reload the picker (browser refresh) and re-enable the agent via the admin whitelist if needed.

Runbook 4: a mode does not answer (wrong LLM Mesh id)

Symptom: a specific mode fails systematically (often Eco, which is the default).

Cause: a model id in LOOP_LLM_BY_MODE does not match any model exposed by the instance's LLM Mesh connection. Since DEFAULT_MODE = "eco", a wrong GEMINI_FLASH_LITE_ID breaks almost all requests.

Resolution: fix the offending id in both agent files, then re-paste both Code Agents on the Python 3.11 env.

In Flux - model ids

The LLM Mesh ids migrated in Run 6 (2026-06-17). They are best-effort in the observed format and must be re-verified against the instance on every re-paste. A wrong id breaks the corresponding mode silently.

Runbook 5: slow answers in High mode

This is not necessarily a failure. High drives Claude Sonnet 4.6 end to end; Sonnet reasons longer. The polling transport does not stream word-by-word: the prose arrives in one block at the end. If latency exceeds ~5 min, MAX_RUN_SECONDS = 300 s cuts the run off (run_timeout). For speed, advise Eco (Gemini Flash-Lite, the default) or Medium (Gemini Flash).

Runbook 6: empty chart in Evidence

Cause: an artifact spec (show_chart / show_table) carries only the display spec, never the data rows. The Chart.js payload is built from the captured result of generated_sql. No captured result = empty chart. result_captured=false is a normal best-effort limitation.

Resolution: if every chart is empty since an agents edit, verify the tool span is still called semantic-model-query in both agent files and re-paste both.

Runbook 7: non-clickable Evidence source

Cause: the source_url field on the revenue_expert capability of the orchestrator registry defaults to an empty string. Fill it in (OWIsMind_orchestrator.py) with the Dataiku dataset URL, then re-paste both agents.

Runbook 8: user blocked by monthly budget (HTTP 402)

Symptom: a user sees a banner indicating the budget is exhausted; all /chat/start calls return HTTP 402 monthly_quota_exceeded. Reading and Evidence still work.

Resolution: from the admin Quotas tab, choose one of:

  • Per-user override (permanent or temporary, via POST /admin/budget/users)
  • Global temporary boost (POST /admin/budget with temp_limit_usd)
  • Raise the global default (POST /admin/budget with limit_usd)
  • Disable enforcement entirely ({enabled: false} - spend is still recorded)

The budget auto-resets on the 1st of each calendar month (new DB row, no manual action).

Runbook 9: agent profile shows "profile to complete"

Cause: no admin has filled in the profile yet. This is the intended post-deploy state - there is no hardcoded fallback. Navigate to Administration > Agents > Edit profile, fill in all fields (tagline, description, capabilities, tools, icon, badge) and save. The change is effective immediately; no restart needed.

Testing

OWIsMind is tested in three places, one per layer. All three suites share the same philosophy: pure-logic, run with the language's native runner, without a DSS environment and without any installation. They lock down the invariants testable off-instance but do NOT replace validation in DSS.

NO INSTALL - non-negotiable rule #1

The agent never installs a dependency. The commands below use only python3 and npm with built-in runners. If node_modules/ is missing for the build, the user installs, not the agent.

Agents suite (267 tests) and the anti-drift test

Stubs dataiku AND langgraph before the agent files are loaded, so only pure logic is exercised. The graph itself is NOT executed (needs DSS). The most important test is the anti-drift test:

Anti-drift invariant

The block_labels and tool_labels declared in the orchestrator registry (CAPABILITIES) must exactly match the KNOWN_BLOCK_IDS and KNOWN_TOOL_NAMES of the sub-agent. If someone renames an event, adds a tool, or changes the sub-agent id without keeping both files synchronized, this test breaks. It is what makes re-pasting both Code Agents safely possible: orchestrator-to-sub-agent consistency is verified locally before deployment.

What requires DSS (not covered by the suites)

Some backend modules import dataiku or pandas at load time and are therefore not covered off-instance. Validation in DSS covers:

  • Real LLM Mesh calls and per-mode model behavior
  • PostgreSQL SQL execution (actual storage)
  • Evidence panel rendering (chart, table, KPI)
  • The actual execution of the LangGraph graph
  • Orchestrator routing decisions (LLM decisions)

Roadmap: TEST-01

Recommended but not yet done: add DSS-free tests based on a dataiku/pandas stub for the already-hardened but uncovered invariants (sql_config.pg_identifier, stream_manager, resolve_enabled_agent). A minimal CI pipeline: lint + py_compile + backend unittest + vite build. There is NO CI today.

Agent evaluation and golden queries

Beyond the pure-logic suites, agent quality is evaluated in DSS via the DSS Agent Review. Both agents open named sub-spans via trace.subspan(...) that feed both the Agent Review and the webapp Evidence panel. Key spans:

  • Orchestrator: orchestrator:llm, semantic-model-query
  • Sub-agent: dataset-expert:understand, dataset-expert:resolve-values, dataset-expert:semantic-tool, dataset-expert:sqlgen, dataset-expert:headline, and one semantic-model-query per executed SQL (with {sql, success, row_count})

The Semantic Model has 9 golden queries (versioned in tools/semantic_model/build_aligned_semantic_model.py) that serve as few-shot examples and accuracy reference. The canonical pitfall case: "YTD revenue EVPL, actuals vs budget" - EVPL exists at Product, Solution and sirano_product levels; a naive pin on sirano_product gives budget = 0 (BUDGET rows may not carry it). Golden query 4 anchors the correct resolution at the Product level.

Known gotchas and lessons

A synthesis of the cross-cutting gotchas that genuinely cost time. Each comes from a real failure in DSS or review. Reintroducing a banned pattern replays a bug already paid for.

Frontend: key gotchas

F-3: :global theme selector must wrap the ENTIRE selector: :global(body[data-theme="dark"] .x), not just :global(.x).

F-4: hardcoded rgba colors are invisible in dark mode. Use semantic tokens from styles/tokens.css. No color-mix.

F-8: the real logo (orange-logo.png) must always be used as an <img>. Never reconstruct it as a CSS shape. Rule #10.

Backend: key gotchas

B-1: no ALTER TABLE ever. Schema evolution = new _vN table via CREATE TABLE IF NOT EXISTS. Current chat table = webapp_chat_v5.

B-5: after any python-lib/ or backend.py change, RESTART the webapp backend in DSS. The old code runs until restart.

B-6: the cp to body.html may be refused by harness permissions. Fallback: rewrite via the Write tool (gotcha F10).

Agent: key gotchas

A-1: never use as_langchain_chat_model. Always call LLM Mesh natively in the nodes (reasoning + tool-calling preserved).

A-3: force with_json_output on any output consumed by code (UNDERSTAND step). Reasoning reserved for routing and verified prose only.

A-8: the orchestrator emits zero business facts. The only "no" allowed = "no AGENT yet for this domain". When in doubt, ROUTE. Never assert "the data does not exist".

Packaging: key gotchas

P-1: frontend/ and node_modules/ must NEVER be packaged. Only runtime is staged: plugin.json + python-lib/ + resource/ + webapps/.

P-2: NEVER build into resource/ outside /build-plugin. emptyOutDir: true would overwrite the deployed app.

P-5: em dash (U+2014) and en dash (U+2013) are banned everywhere. The sweep must be byte-safe (LC_ALL=C; never perl -CSD on files with multibyte glyphs like owi:mode tokens).

Four reflexes that prevent the majority of regressions:

  1. When you change python-lib, RESTART the webapp backend - otherwise your fix does not run.
  2. When you change a Code Agent, RE-PASTE both files into DSS (env 3.11) - the repository is the source of truth.
  3. Never reintroduce a banned pattern: hardcoded color, ALTER, as_langchain_chat_model, narrate-first, em dash, CSS-generated logo.

The memory protocol

OWIsMind keeps a project memory under memory/ that takes precedence over the guides in docs/cadrage/. Three files structure it:

memory/CONTEXT.md
Short-term memory (current focus, last session, active gotchas). Read at EVERY session start.
memory/LESSONS.md
L0xx lessons: what diverged from the guides, what failed then worked. Read when looking for the "why" of a non-obvious solution.
memory/PROJECT_STATE.md
Durable state: architecture, canonical ids, validated/not-validated matrix in DSS.

At the end of each session, run the /log-session skill: it writes memory files, appends a lesson if there is a divergence, updates the knowledge graph (graphify --update), and creates a session commit. It never pushes (the user pushes).
