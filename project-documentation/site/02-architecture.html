The Four Layers

OWIsMind's architecture reads as four stacked layers connected by deliberately narrow contracts. Each layer knows only what it needs from the one below. This diagram is the canonical home of the system context: other documents refer back to it.

Source: 02-architecture/01-system-overview.md

Layer descriptions

Vue 3 SPA (frontend)

Where: Plugin/owismind/frontend/, built into resource/owismind-app/

Renders three screen areas: conversations sidebar, chat thread with live timeline, and the Evidence Studio panel. Sends only opaque logical keys to the backend - never a raw agent_id, table name, or SQL. Polls /owismind-api/chat/poll every ~500 ms while a run is live.

Flask backend (python-lib)

Where: Plugin/owismind/python-lib/owismind/ - Python 3.9.23

A Flask Blueprint mounted under /owismind-api. Resolves identity, applies the agent whitelist, launches the agent in a daemon worker thread, persists in direct SQL, and re-derives Evidence purely deterministically. No langchain, no langgraph.

LangGraph agent layer (Code Agents)

Where: dataiku-agents/agents/, pasted into DSS Code Agents - Python 3.11

OWIsMind_orchestrator routes to the revenue expert sub-agent SalesDrive_revenue_expert (agent:bHrWLyOL). Both use LangGraph with native LLM Mesh calls. The orchestrator holds the honesty firewall: it never holds a business figure. The repo is the source of truth; agents are pasted by hand, not zipped.

PostgreSQL storage (direct SQL)

Where: SQL_owi connection, schema public

Six tables in _vN naming, created lazily on first use, never ALTER-ed structurally. All accessed via SQLExecutor2 with parameterized values and explicit COMMIT. No Flow at runtime (the only exception is the write-only trace dataset).

Guiding principles

Five cross-cutting invariants explain why the layer boundaries are where they are. Each is backed by an architecture decision record (ADR).

1 - Direct SQL, no Flow at runtime

All application state (conversations, messages, feedback, user registry, settings, token usage, artifacts) is persisted in direct SQL via SQLExecutor2 on the admin-configured SQL_owi connection. Requires: a fresh executor per call (never shared across threads), physical naming {PROJECT_KEY}_{namespace}_{logical}, _vN versioning (no ALTER), explicit COMMIT, always-parameterized values, systematic user_id scoping. The only exception: the write-only trace dataset, appended via dataiku.Dataset.write_with_schema to avoid inlining a large JSON blob in a logged INSERT.

ADR-0003 SQLExecutor2 parameterized

2 - Server-side agent whitelist, opaque logical key

The frontend never picks an agent in the clear. It sends an opaque logical key of the form ag_<sha1[:12]>; the backend resolves it into (project_key, agent_id) via settings.resolve_enabled_agent. A forged or disabled key resolves to None and can never reach an executable agent. The whitelist is two-tiered: the backend resolves the orchestrator; the orchestrator's CAPABILITIES registry resolves the sub-agent by capability name (e.g. ask_revenue_expert), never by raw id from the front.

ADR-0004 agent_key opaque never raw agent_id

3 - Repo = source of truth for the agents

The two Code Agents are NOT packaged in the plugin zip. They live in dataiku-agents/agents/ (source of truth) and are pasted by hand into DSS Code Agents on a Python 3.11 env (where langchain/langgraph are installed). The backend runs on Python 3.9.23 (Flask, no langchain). When either file changes, BOTH are re-pasted together - some fixes live on both sides. Configuration ids are re-verified after pasting.

ADR-0005 Python 3.11 outside the zip

4 - Signal vs data: the orchestrator holds no figure

The orchestrator NEVER holds a business figure: every number comes from a sub-agent (SQL-grounded) so it structurally cannot invent one. The "honesty firewall" manifests as: (a) an artifact event carries only a display SPEC ({kind, title, chart|kpi}), never the rows - the data is the separately captured result; (b) the live polling timeline strips the result key - data is read only at persistence; (c) grounding on value_index is read-only inline SQL in the sub-agent, NOT a tool - the labels resolve_filter_value and dataset_sql_query on the timeline are event names, not tool calls. The only real DSS tool called at runtime is revenue_semantic_query (v4oqA6R).

ADR-0008 ADR-0010 signal / data separation

5 - Dataiku instance safety, everywhere

Every code path first asks: is this risky, slow, or overloading for the instance? This translates into explicit bounds: MAX_CONCURRENT_RUNS = 8, MAX_RUN_SECONDS = 300.0, TTL eviction of orphaned runs, MAX_PERSISTED_TEXT_CHARS = 262_144, caps on captured results, MAX_CHAIN_DEPTH = 200 for the ancestor tree, MAX_PARALLEL_AGENTS = 3 for sub-agent fan-out. No generic SQL route is exposed; the frontend never picks a table, connection, or query.

concurrency capped at 8 300 s wall-clock limit no generic SQL route

Transport: polling, not SSE

The transport between the SPA and the backend is polling via a background thread, NOT Server-Sent Events. DSS places an internal nginx in front of each webapp backend; that nginx buffers a long text/event-stream response, so all events would arrive at once at the end. OWIsMind works around this exactly like the production Dash app on the same instance: the agent runs in a daemon worker thread, its progress accumulates in a module-level dict (_RUNS), and the frontend polls that dict with short requests the proxy never buffers.

Visible consequence

The text answer often lands in a block at the end of a run. What is genuinely usable live is the execution timeline, not word-by-word streaming. This is a platform constraint, not a design limitation.

The artifact flow (show_chart to Chart.js)

The system rigorously separates the SIGNAL (what the orchestrator asks to display) from the DATA (the real rows). An ARTIFACT event carries only a SPEC ({kind, title, chart|kpi}), never the rows. The Chart.js payload is rebuilt server-side at read time from the captured generated_sql result.

Artifact pipeline, step by step

  1. Orchestrator calls show_chart(chart_type, x, y) - validated against exact columns of the last result.
  2. agents/streaming.py normalizes the ARTIFACT event to a strict form: kind in {chart, table, kpi}, bounded title, y bounded to 8 series.
  3. Worker accumulates the spec (cap MAX_ARTIFACTS_ACCUM = 8), NOT on the live timeline.
  4. At end of run, save_artifacts writes an owner-stamped UPSERT to webapp_artifacts_v1. Best-effort: a failure never aborts the response.
  5. When the Evidence panel opens, /evidence/meta reads the spec and calls chart_payload.build_chart_payload(result_block, chart_spec) server-side. A mistyped column or non-numeric cell degrades to an honest empty state ({ok: false, reason: ...}), never a broken chart.
  6. ArtifactChart.vue renders the already-secured Chart.js payload. This is the culmination of signal/data separation.

In Flux - Evidence result capture

The key of the tool span's rows is not confirmed on the instance. The capture of the Evidence result is best-effort: it may be absent (result_captured: false), in which case the chart degrades to an empty state but the rest of the Evidence panel remains useful.

Security Model

OWIsMind is a multi-user WebApp served by a shared Dataiku DSS instance. Security does not rest on "who can reach the app", but on four pillars: identity, agent whitelist, SQL safety, and owner-scoping. The trust boundary is the browser: everything arriving from it is untrusted.

Source: 02-architecture/04-security-model.md

Two identities

This is the most often misunderstood distinction in the system. Two identities coexist at runtime.

Aspect Connected DSS user (caller) Backend run-as-user (WebApp identity)
Origin Browser auth headers (DSS session cookie) Identity under which the DSS WebApp backend process runs
Resolution resolve_identity(request.headers) - 5 s TTL cache Implicit: the DSS execution context for SQLExecutor2, LLM Mesh, and discovery
Key value user_id = info.get("authIdentifier") (DSS login) Never appears as a value in code; it is the execution context
Role Scoping ALL storage queries (owner-scoping); application admin election Actually runs SQL and calls agents; project/agent visibility reflects its DSS permissions

TOFU - first user is admin

The first user to open the app after configuration becomes the application admin (Trust On First Use, via pg_advisory_xact_lock). On deployment, ensure it is the intended admin who opens the app first. The application admin flag (is_admin in webapp_users_v1) is distinct from DSS admin rights.

Hash router: why

Vue Router runs in HASH mode (createWebHashHistory) because DSS serves the webapp at a fixed URL with no server-side SPA rewrite. A path-based history would yield a 404 on reload or deep-link. See ADR-0001.

Dual Python environment (3.9 and 3.11)

The dual-environment is a platform constraint, not a choice:

Python 3.9.23 - Flask backend

The DSS webapp Python environment, observed in production via /owismind-api/ping. Flask is provided by DSS via from dataiku.customwebapp import *. The backend does NOT bundle langchain or langgraph: it only uses Flask plus direct SQL. Rule #8: do not assume Python 3.11 or FastAPI work here.

Python 3.11 - LangGraph agents

A separate DSS code env where langchain and langgraph are installed. LangGraph / LangChain v1 require Python >= 3.10 and therefore cannot live in the backend's 3.9 env. The agents are pasted by hand (not zipped). Their import stack is stdlib + dataiku + langchain/langgraph only: zero import of the plugin code.

In Flux - per-mode LLM Mesh ids

The ids declared at the top of each agent file (GEMINI_FLASH_LITE_ID, GEMINI_FLASH_ID, SONNET_ID) must match an id actually exposed by the instance's LLM Mesh connection. A wrong id breaks the corresponding mode. They must be re-checked in DSS after each re-paste of the Code Agents. The Semantic Model Query tool (v4oqA6R) always uses Sonnet regardless of the loop mode.

The NO INSTALL rule

NO INSTALL is the project's structuring rule #1: the agent (the assistant) NEVER installs a dependency - no npm install, no pip install, no brew, no npx. Only the user installs. This rule is enforced at three levels: harness permissions (.claude/settings.json deny list), a pre-tool hook (guardrail.sh that blocks install commands), and documentation.

The non-trivial consequence: resource/owismind-app/ (the built frontend) is git-tracked, even though it is a regenerated output. Because a fresh clone cannot reinstall the toolchain, the plugin payload must travel within the repository to remain packageable. It must NEVER be edited by hand.
