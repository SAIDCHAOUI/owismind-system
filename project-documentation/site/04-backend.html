Overview and structure

The OWIsMind backend is a standard DSS WebApp with a Python Flask backend (hasBackend: "true" in webapp.json). It acts as an intermediary between the Vue 3 frontend on one side and the LLM Mesh agents plus the PostgreSQL database on the other. It never reasons on its own: it validates, routes, persists, and normalizes events.

Observed runtime

The backend runs on Python 3.9.23. No FastAPI, no langchain. LangGraph lives in the Code Agents in a separate 3.11 env, outside the backend zip. Do not assume Python 3.11 features are available here.

The thin bootstrap

DSS injects the Flask app object via a star-import. The webapp's backend.py contains no logic - it only wires the plugin's routes:

webapps/webapp-owismind-ai-agents/backend.py Python 
from dataiku.customwebapp import *   # provides the Flask app object
from owismind.api.routes import register_routes
register_routes(app)

register_routes(app) does four things at boot: (1) mounts the blueprint; (2) applies the admin log level; (3) logs the resolved storage_status(); (4) logs the live route table (a landmark in the DSS backend log confirming which build is running and how many routes are mounted). Steps 2 and 3 are wrapped in a try/except so that an unreadable storage config cannot prevent the app from starting.

Sub-packages of python-lib/owismind/

All logic lives in python-lib/owismind/, split into five thematic sub-packages. The dependency direction is fixed: api/ calls everything; agents/ and evidence/ call storage/; nothing calls api/ from below.

api/

Single blueprint in routes.py: blueprint-scoped before/after hooks (meta only, never message content), guards (_evidence_guard, _admin_guard), and all 21 handlers. No generic SQL route - the frontend never chooses a table, connection, or query.

security/

identity.py: resolves caller from DSS auth headers, short TTL cache for poll load. validation.py: pure validators (no DSS dependency), stable error codes, NEVER trusts the frontend body.

agents/

stream_manager.py: run lifecycle, _RUNS dict, worker threads. streaming.py: native LLM Mesh call, event normalization. context.py: pure multi-turn payload builder. discovery.py: read-only project/agent listing for the admin picker.

evidence/ and storage/

Evidence re-derives verification deterministically from stored SQL (zero LLM). Storage persists all state in direct SQL via SQLExecutor2 - no Flow at runtime (single exception: the write-only trace dataset).

Streaming and runs

OWIsMind does not push a long SSE stream. DSS places an internal nginx in front of every webapp backend: a long text/event-stream response would be buffered by the proxy and arrive all at once at the end. The chosen pattern (also used by the production Dash WebApp on the same instance) is: run the agent in a background daemon thread, accumulate progress in a module-level dict (_RUNS) guarded by a single threading.Lock, and let the frontend poll every ~500 ms. Each poll is a short request the proxy never buffers. This choice is formalized in ADR-0002.

No LLM Mesh cancel API

The stop is purely cooperative (ceasing to iterate the generator), evaluated between chunks. An upstream LLM call that is fully blocked is bounded only by the HARD_TTL eviction. A watchdog thread was deliberately not added (higher risk on a valid path). The official LLM Mesh stream exposes no cancel method.

Storage and data model

The backend persists ALL application state in direct SQL via SQLExecutor2 on a PostgreSQL connection (schema public). No Flow at runtime (single exception: the trace dataset, write-only). Every call creates a fresh SQLExecutor2 instance. Writes always include an explicit COMMIT. All values are parameterized via sql_value().

The six tables (_vN naming)

Logical namePurposePK
webapp_chat_v5One chat exchange per row. Written in two phases (user message first, assistant reply on run completion). Carries 4 usage columns: authoritative per-exchange token/cost record.exchange_id
webapp_users_v1User/admin registry. Upserted on every POST /me. Holds lifetime token/cost counters (denormalized accelerator, reconstructible from webapp_chat_v5).user_id
webapp_settings_v1Global key-value config. Carries the agents whitelist (key enabled_agents) and global budget config (key monthly_budget). NOT per-user.setting_key
webapp_usage_monthly_v1Monthly usage bucket per (user, calendar month). Composite PK guarantees exactly one row per month without a reset job. Incremented by UPSERT on each run completion.(user_id, period_start)
webapp_artifacts_v1Artifact specs (chart/table/kpi) per exchange. Only the SPEC is stored; data rows are reused from the captured generated_sql.result via /evidence/meta.exchange_id
webapp_user_quota_v1Per-user monthly budget overrides. A row exists ONLY for users with a custom limit; absent row = global default applies. Created lazily on first write. Added 2026-06-18.user_id

_vN strategy: no structural ALTER

A new row format = a new _vN table (the old one is left inert, never dropped by the backend; old conversations simply stop surfacing). The only authorized relaxation: additive ADD COLUMN IF NOT EXISTS on webapp_users_v1 for the lifetime usage counters (user-authorized 2026-06-11). Chat table history: v2 (+generated_sql), v3 (+feedback), v4 (+parent_exchange_id), v5 (+4 usage columns). Always reference webapp_chat_v5, never v4.

SQL data model and conversation tree

Conversation tree

parent_exchange_id turns the flat webapp_chat_v5 into a tree. Each exchange points to the exchange it branched from; NULL = root. Editing or regenerating a prompt creates a sibling exchange. The agent context of an exchange = its ancestors chain only: the backend walks up the parent_exchange_id chain from the parent to the root of this branch via a recursive CTE. A branch never sees messages that came after its branching point, nor other branches. The CTE carries user_id in BOTH the anchor and recursive members (owner-scope in both), bounded to depth 200 (anti-cycle).

3-level usage accounting

LevelTableGranularityWrite
1 - source of truthwebapp_chat_v5 (4 usage cols)Per exchangePhase-2 UPDATE (atomic with assistant_text)
2 - lifetime totalwebapp_users_v1Per user, all-timeAdditive UPDATE in record_usage()
3 - monthly bucketwebapp_usage_monthly_v1Per (user, month)Incrementing UPSERT on composite PK

Levels 2 and 3 go out in one committed transaction (pre_queries=[monthly_sql, users_sql], post_queries=["COMMIT"]). Best-effort by contract. Level 1 is the only authoritative record; levels 2 and 3 are reconstructible by summing webapp_chat_v5.

Monthly budget resolution (storage/budget.py)

storage/budget.py is the single source of truth for the monthly limit math. Default rolling credit: $50 USD/user. Resolution order (highest priority first):

  1. Active per-user override (webapp_user_quota_v1): row exists AND (expires_at > now() OR expires_at IS NULL). Source tag: user_permanent or user_temp.
  2. Active global temp boost (in webapp_settings_v1, key monthly_budget): evaluated against the app clock. Source tag: global_temp.
  3. Global default (limit_usd in the same JSON). Source tag: default.

An in-process config cache (TTL 30 s, lock-free under GIL, busted on write) keeps the /chat/start hot path to a single DB read. The enforcement gate (has_budget()) fails OPEN by design: a storage error lets the run proceed, and the spend is still recorded for next time.

Agent profile stored in settings JSON

The admin-authored agent profile (tagline, description, capabilities, tools, icon, badge) lives INSIDE the enabled_agents JSON of webapp_settings_v1, under the profile key of each agent entry. No separate table was created. The logical_key is derived from a stable SHA-1 hash of project_key:agent_id (re-saving keeps the same key; existing conversations referencing that key stay valid). The profile is validated/bounded by validate_agent_meta (pure, never raises) before storage.

Evidence and artifacts

Evidence Studio is the panel to the right of the chat. On the backend side, it never asks an LLM to justify an answer: it re-derives everything, in a purely deterministic way (zero LLM), from the single SQL statement the sub-agent stored. The stored SQL is the source of truth; at evidence time, nothing new is written (exception: artifact specs, persisted once at the end of the run). Sources: python-lib/owismind/evidence/ and storage/artifacts.py.

Core principle: separate signal from data

An ARTIFACT event carries only the SPEC (kind, title, and for a chart {type, x, y[]}). It never carries the rows. The DATA is the result already captured on the stored generated_sql item, reused at read time via /evidence/meta. The Chart.js payload is shaped by trusted Python code (chart_payload.py), not by the agent ("the agent only says x/y, the backend builds the chart").

Evidence and artifact pipeline

Trust levels (verification scale)

Computed deterministically by verification_level(). The UI badge can NEVER be green. Dropped or unmapped elements are listed, never hidden.

LevelCriterion
declaredParse fails or no dataset matched. Agent claim only, no verification possible.
source_identifiedDataset matched but WHERE not evaluable, or nothing mapped without completeness.
scope_partialMatched + at least 1 mapped predicate, broken completeness (drops listed).
scope_exactEvery WHERE conjunct decomposed + single source + no set-op.
calc_decomposedscope_exact + SELECT calculation fully understood (nothing opaque in the SELECT list).

result_captured is ORTHOGONAL to the trust level. A failing explainer can only LOWER the level. Drill is offered only when provably reliable: explain ok, no recursive CTE, no set-op, single source, complete reconstructed WHERE, at least one identity GROUP BY key resolved against the live schema. Drill beyond 8 keys is refused, not silently truncated.

sql_parse and sql_explain: pure analysis

sql_parse.parse_select(sql) is best-effort (never fails on complex queries). It recovers source tables and WHERE predicates, marking editable ops (=, IN) and advanced fragments. validate_fragment(text) is the final defensive gate before re-execution: bans DDL words, pg_* functions, backslash, second statements, unbalanced parens; max 2000 chars.

sql_explain.explain_select(sql) produces a structured business explanation. NEVER RAISES (whole body guarded). The honesty principle: anything not positively understood degrades a flag or produces an opaque step. A false explanation would be false evidence; an under-stated one is merely less useful.

In Flux - result capture

The exact key of the tool span rows is NOT confirmed on the live instance. capture.extract_result() probes candidate keys in order: rows, records, data, result_rows, values. An absent capture yields result_captured: false; the panel stays useful (sources, chips, explanation, drill, SQL) but without a chart or a captured result table. This is the single most uncertain point in the whole Evidence area.

Security and validation

The trust boundary is the browser: everything from it is untrusted and must be validated AND bounded before reaching SQL. Two core invariants: (1) the caller's identity is NEVER read from the request body; (2) the frontend only sends logical data - it never chooses table, column, connection, query, or raw agent_id.

Identity resolution (security/identity.py)

resolve_identity(headers) calls dataiku.api_client().get_auth_info_from_browser_headers() with a fresh DSS client per call. The user_id comes from authIdentifier (DSS login). DSS returns no display name, so it is derived: said.chaoui becomes Said (display name) or Said Chaoui (full name, used in the agent context suffix). Failure raises IdentityError mapped to 401 unauthenticated.

A per-process identity cache (TTL 5 s, keyed on SHA-256 of the auth cookie) collapses the repeated resolutions of /chat/poll (~2 Hz per live chat). A different cookie produces a different fingerprint: the cache can never return another user's identity. Only successful lookups are cached.

Payload validation (security/validation.py)

PURE module (no DSS dependency, testable outside the runtime). Every function validates shape and bounds, then raises ValidationError(code) or returns a sanitized/clamped value. Codes are stable strings rendered as-is to the frontend for i18n mapping.

Validator / patternBehavior

Agent whitelist: server-side enforcement

The opaque agent_key is derived as "ag_" + sha1(f"{project_key}:{agent_id}").hexdigest()[:12]. Stable (re-saving keeps the same key) and opaque (frontend never sees the raw agent_id). The enforcement point is settings.resolve_enabled_agent(agent_key) in /chat/start: returns {logical_key, project_key, agent_id, label} ONLY if the key matches a still-active agent, otherwise None - the run is never launched. The resolved agent_id stays server-side end to end.

On the admin write side, POST /admin/agents re-validates each requested agent against the live DSS listings (project visible AND agent actually present) before persisting. A defensive cap (MAX_ENABLED_AGENTS = 50) bounds the selection.

Evidence re-execution runs with SET LOCAL statement_timeout TO '30000' and SET LOCAL transaction_read_only TO on (transaction-scoped: a pooled JDBC connection can never carry these to other workloads). These are pre-queries on the matched dataset's OWN connection, not the chat storage connection.

Admin gating, bootstrap and anti-lockout

All /admin/* routes chain: auth (401) - storage guard (409) - admin.is_admin(user_id) check (403 on false, 500 on exception). The APPLICATION admin (table flag in webapp_users_v1) is distinct from DSS rights.

The first user to issue POST /me after configuration becomes admin. Race-free: a pg_advisory_xact_lock (key 0x4F57494D, "OWIM") serializes the election across concurrent connections (READ COMMITTED isolation would otherwise let two concurrent users both see "no admin"). GET /me is read-only (never creates a row or wins the election).

TOFU - first to open = admin

The first user to open the app after configuration becomes admin. At deployment, ensure this is the intended deploying admin before anyone else opens the webapp. Anti-lockout guard in POST /admin/users/set-admin prevents removing the last admin (400 cannot_remove_last_admin).

Log hygiene

Message content is NEVER logged. The blueprint-scoped hooks trace only meta (method, path, status, duration in ms) for /owismind-api/* only (not DSS internal pings). Header values (credentials) are never logged. On an unexpected auth shape, only the KEY NAMES (sorted(info.keys())) are logged, never the values. /ping deliberately does not expose the storage config.

In Flux - single-process assumption

All in-memory state (_RUNS, identity cache, rate-limit buckets, Evidence caches, budget config cache) assumes a SINGLE-PROCESS backend. In multi-process, the concurrency cap would be multiplied and cross-process poll would return 404. Force 1 process at deployment. The first-admin election stays protected by the PostgreSQL advisory lock (cross-connection, not process state).
