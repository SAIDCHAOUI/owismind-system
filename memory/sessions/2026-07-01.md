- **Cause** : `/api/golden` (`views.golden_tag_view`) renvoyait `{rows}` sans `agents`, scope `this` ;

- **Fix** : `golden_tag_view` renvoie `questions` (+ scope `agent`, alias `this`) ; la route ajoute

- User : "ça marche beaucoup mieux".

### 3. agent_key = l'ID de l'agent PARTOUT (jamais un slug) - ✅ VALIDÉ DSS
- **Directive user** : « jamais orchestrator, toujours l'id de l'agent ; le nom c'est juste pour
  l'affichage ; tout géré depuis la webapp (on récupère les agents + leurs id, on clique, pas de
  hardcode) ». 3 clés divergeaient pour le même agent : slug `orchestrator` (variable/exemple/MOCK),
  slug projet dérivé `owismind_dev_038g7mlf` (connect flow), et l'id `038G7mlF` (le golden). Le golden
  avait raison.
- **Fix** : `registry.agent_catalog_key(project, agent_id)` = **agent_id sans préfixe `agent:`,
  verbatim** (drop du slug + prefixe projet). `registry.normalize_agent` **dérive toujours** la clé de
  l'id (ignore toute `agent_key` hardcodée). `run_params._resolve_agents` idem (dérive l'id, ignore la
  clé, ne drop plus sur clé vide mais sur project/agent_id vides). Purge des `orchestrator` en dur :
  `local-variables.example.json`, MOCK launcher (`agent_key`/`agent_tagged`), MOCK results, docstrings
  `run_params.py`, `SETUP_GUIDE.md`. Tests registre + run_params mis à jour (clé = id).
- Les fixtures de tests gardent `orchestrator` comme chaîne opaque (aucun impact).

### 4. Formulaire golden : 3 colonnes manquantes (édition de TOUT le dataset) - non validé DSS

  colonnes y étaient déjà).

### 5. Fix redo `value`/`include_next` ("nothing to run" au redo) - non validé DSS
- **Cause** : `/api/benchmark/redo` lisait `body.get("include_next")` ; le front (et le MOCK) envoient
  **`value`** -> `bool(None)=False` -> **chaque clic "à refaire" = décocher** -> le flag n'est jamais
  stocké -> `resolve_to_run` ne voit aucun redo -> `ValueError: nothing to run for benchmark ...`.
- **Fix** : le backend lit `value` (`include_next` en alias défensif). Scan des autres endpoints
  benchmark (create/delete/modes/launch) = **aucun autre décalage** de nom de champ, `redo` était le seul.
- **Défaut secondaire trouvé (non corrigé, proposé)** : `reconcile_redo_after_run` efface le redo des
  questions du **dernier run scored existant** sans vérifier que ce run a bien consommé le flag (postérieur
  au flag). Sur un run qui échoue / ne produit aucune ligne scored, il nettoie le redo sur l'ANCIEN run.
  Le chemin normal (run réussi) marche. Fix propre = snapshot du dernier `run_timestamp` au launch +
  ne nettoyer que les questions dont une ligne scored est plus récente. En attente d'accord user.
