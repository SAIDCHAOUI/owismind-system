## Run 2 — SSE bufferisé EN DSS → bascule POLLING-via-thread (✅ VALIDÉ DSS) + 2 fixes front

### Objectif
Le run 1 (SSE) testé en DSS : **réponse + SQL OK mais aucun stream live** — tous les eventKind + la réponse
arrivent **en un bloc à la fin** (« j'attends j'attends puis tout d'un coup »). Confirmation du **buffering proxy
DSS** (L005/L018). Objectif : avoir la **timeline qui défile en direct**, sans nuire/ralentir l'instance.

### Démarche (importante)
Plutôt qu'un pattern théorique, on est partis du **Dash de prod du client** (placé dans `old_webapp_in_dash/`) +
recherche doc officielle. **3 agents** lancés (2 lecture code Dash, 1 recherche web). Verdict convergent :
- Le **Dash de prod** (même instance DSS) **ne fait PAS de SSE** : il lance le run agent dans un **`threading.Thread`
  daemon**, accumule la progression dans un **dict module-level + `Lock`**, et le front **poll ce dict via
  `dcc.Interval` à 250 ms**. Il **n'expose jamais de réponse HTTP longue** → contourne le buffering **par design**.
  Sa « timeline agents working » est **cosmétique** (messages JS en dur), PAS de vrais events.
- Doc : Dataiku **Answers** fait du SSE en interne, mais **aucun exemple officiel de SSE depuis un backend de
  webapp standard** ; DSS met un **nginx interne devant chaque backend** → `X-Accel-Buffering` non garanti.
- **Décision user** : « Go, remplace le SSE » par le **polling-via-thread** (pattern prouvé en prod), + garde-fous
  que le Dash n'a pas. (User réticent au thread/RAM au départ → levé par le constat que SA prod le fait déjà.)

### Fait (changements concrets)
**Backend**
- **`agents/stream_manager.py`** (NOUVEAU) — cœur du polling : `_RUNS` dict + `_LOCK`, **worker daemon par envoi**
  qui itère `run_agent_streamed` (INCHANGÉ, réutilisé tel quel), empile les events normalisés, accumule
  réponse/SQL, persiste l'assistant (phase 2), marque `done`. **Garde-fous absents du Dash** :
  `MAX_CONCURRENT_RUNS=8` (cap, 503 `busy`), **TTL** `FINISHED_TTL=60s`/`HARD_TTL=600s` (éviction, zéro fuite),
  **scope par `user_id`** (un run pollable que par son owner). Émet la MÊME séquence d'events que le SSE
  (`run_started`/agent_event/answer_delta/generated_sql/usage_summary/final_answer/run_done/error). `done` posé
  **après** les events terminaux, lecture slice+done **sous 1 lock** → pas de perte de frame finale.
- **`api/routes.py`** — `/chat/stream` (SSE) **SUPPRIMÉ** → **`/chat/start`** (POST : valide, résout agent, save
  user phase 1, `stream_manager.start_run`, renvoie `{run_id, exchange_id}` ; 503 `busy` si cap) + **`/chat/poll`**
  (GET `?run_id=&cursor=` → `{events, cursor, done, error}` ; 404 `run_not_found` si inconnu/autre user). Retiré
  `_sse`/`_SSE_HEADERS`/`Response`/`stream_with_context`/`json`/import `streaming`.
- **`security/validation.py`** — `validate_chat_stream_request` → renommé **`validate_chat_start_request`**.
- `agents/streaming.py` et `storage/chat_v2.py` : **INCHANGÉS** (normalisation + écriture 2-temps réutilisées).

**Frontend**
- **`services/backend.js`** — `streamChat`/`emitFrame` (reader SSE) **SUPPRIMÉS** → **`startChat`** + **`pollChat(runId,
  cursor)`** (via le helper `request`).
- **`App.vue`** — `send()` : `startChat` puis **boucle de poll à `POLL_INTERVAL_MS=500`** jusqu'à `done` ;
  `handleEvent` **identique** (seul le transport change). **FIX réactivité** : `assistant` passé par **`reactive()`**
  (sinon mutations via réf brute → 0 re-render → tout s'affiche à la fin). **FIX flexbox** dans `App.css` :
  **`.msg { flex-shrink: 0 }`** (sinon, en colonne flex scrollable, les enfants se compressent au lieu de scroller
  → bulles user écrasées/cachées derrière les voisines).

**Build/package** — 3 cycles Vite (chaque fix). Hash finaux **`index-CqEmZ-DJ.js` / `index-BnW-lwJY.css`**. `body.html`
recâblé via **Edit** (cp toujours refusé). Zip **39 fichiers, propre** (`stream_manager.py` ajouté, `chat_v1` absent).

### Validé / NON validé
- ✅ **VALIDÉ EN DSS** (logs backend partagés par l'user) : `/chat/start` → worker lancé → **`/chat/poll` toutes les
  ~600 ms en 3-4 ms** (zéro buffering) → **curseur qui avance PENDANT le run** (`0→1→3→5→6→7→14`) = events livrés au
  fil de l'eau. `generated_sql` (sql_count=1), réponse persistée `chat_v2`, **multi-agents OK** (2 agents testés :
  `agent:0GePTo1X`, `agent:ZiaxbfQa`). Après fix réactivité : **timeline qui défile live** confirmée par l'user.
  Après fix flexbox : **bulles user ne disparaissent plus au scroll** confirmé par l'user.
- ⚠️ Nuance attendue & acceptée : la **réponse texte tombe en bloc à la fin** (agent structuré, bloc `render_answer`)
  — le live exploitable = la **timeline**, pas un typing mot-à-mot.

### Décisions
- **Polling-via-thread** (vs SSE) : prouvé en prod sur la même instance, garanti non-bufferisé, + cap + TTL + scope
  user. Réponse 100% : un seul chemin (SSE supprimé, pas de fallback à maintenir).
- Cadence poll **500 ms** (vs 250 ms × 2 callbacks du Dash) → moitié moins de charge.
