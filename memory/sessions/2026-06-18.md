# Session 2026-06-18 - Resolver rapide `attribute_lookup` + suppression totale de `dataset_lookup`

## Objectif
Deep-dive de l'archi tools + SQL (demande user), puis regler le vrai point de douleur DSS : les
questions simples ("quel est l'account manager de X ?") sont LENTES (semantic model ~80s) et echouent
("pas dans les donnees" alors que la colonne existe), et l'orthographe des colonnes casse tout.
Construire un vrai resolver rapide, le rendre universel, et retirer l'ancien `dataset_lookup`.

## Fait
- **Nouveau Custom Python tool** `dataiku-agents/tools/attribute_lookup_tool.py` (`BaseAgentTool`,
  stdlib + `dataiku`, env 3.9/3.11 OK). Comportement = recherche "Whole data" Dataiku :
  - `ILIKE` sur **toutes les colonnes texte** du fact (insensible casse + accents, needle accent-strippe).
  - Sortie **`found_in`** = la/les colonne(s) ou le terme apparait + sa/ses valeur(s) exacte(s) (le
    "resolver" : ou est la valeur + orthographe exacte). `attributes` (optionnel) -> valeurs des
    colonnes demandees pour les lignes matchees. Statuts : `found` / `suggestions` / `not_found` /
    `attribute_unknown` / `bad_input`.
  - **Fallback alias** optionnel sur `DRIVE_Revenues_Value_Catalog` (suggestions "vouliez-vous dire")
    quand rien ne matche ; le catalog est OPTIONNEL (un catalog absent ne casse rien).
  - **Securite** : read-only (`statement_timeout` + `transaction_read_only`), `LIMIT` partout, rien
    charge en RAM ; seuls des noms de colonnes valides du schema live entrent dans le SQL.
  - **Zero nom de colonne metier en dur** (regle P3) : colonnes lues du schema live ; priorite par
    signaux generiques du catalog (`is_alias`, `frequency`, ordre des domaines). Reutilisable sur un
    autre dataset en changeant 2 lignes de CONFIG.
  - **33 tests** `dataiku-agents/tests/test_attribute_lookup.py` (helpers purs + flux `invoke` avec
    `dataiku` simule).
- **`dataset_lookup` ENTIEREMENT supprime** (pas desactive) de tout le systeme :
  - Sous-agent `SalesDrive_revenue_expert.py` : config `DATASET_LOOKUP_*`, intent `lookup`,
    `build_lookup_filter` / `extract_lookup_rows` / `lookup_note` / `_lookup_rows`,
    `Profile.match_attribute` / `attribute_columns` / `live_columns`, `_read_live_columns`,
    `HEADLINE_LOOKUP`, le chemin LOOKUP de `n_query`, le bloc prompt UNDERSTAND (attributes + intent
    lookup), et les contrats `KNOWN_BLOCK_IDS` / `KNOWN_TOOL_NAMES`.
  - Orchestrateur `OWIsMind_orchestrator.py` (contrat anti-derive lie) : `block_labels["lookup"]`,
    `tool_labels["dataset_lookup"]`, `_BLOCK_NARR["lookup"]`, narration `_NARR["lookup"]`.
  - Tests : 18 tests lookup retires (`TestLookupFilter`, `TestExtractLookupRows`, `TestLookupRows`,
    `TestLookupUnderstanding`, `TestLiveSchemaAttributes`, `TestLookupFilterAmbiguousColumns`,
    asserts KNOWN, narration bilingue).
  - Doc alignee : `dataiku-agents/CLAUDE.md`, README maitre, `agents/README.md`, `tools/README.md`.
- **Verifs** : `grep dataset_lookup` sur code agents + tests = NONE ; les 2 agents parsent (`ast.parse`) ;
  **242 tests verts** (`python3 -m unittest discover -s dataiku-agents/tests`), 0 regression.

## Decisions
- **Resolver = recherche SQL full-text directe sur le fact (ILIKE), PAS via le catalog.** Cause : le
  catalog n'indexe pas certaines colonnes (`account_manager`) -> "blanchard" introuvable via catalog.
  Le full-text couvre TOUTES les colonnes et trouve "blanchard", "indirect", "Telesat", etc.
- **SQL gagne sur le pandas-en-RAM** a 170K lignes (catalog) : charger 170K x 22 = ~0.5-1 Go RAM +
  fuzzy CPU lourd sous 50 users. "tool vs inline" n'est PAS un debat de charge SQL (un tool qui
  requete fait le meme SQL) ; le vrai levier = ou se fait le matching (base vs RAM) -> base.
- **Sortie minimale "resolver"** : `found_in` par defaut ; les valeurs des autres colonnes seulement si
  demandees. (L'user a iterativement recadre : pas une fiche complete verbeuse, juste ou + valeur.)
- **Ambiguite = noms vraiment differents seulement.** Meme nom vu comme compte ET groupe -> meme
  entite -> on prend le plus precis et on repond (plus de fausse ambiguite).
- **`dataset_lookup` supprime completement** (zero code mort, exigence user explicite), pas juste un flag.
- **Branchement DIFFERE** : la conception finale du tool + le point de branchement (sous-agent vs
  orchestrateur) seront tranches la prochaine session via un workflow multi-agents (user = decideur).

## Valide / non valide
- **Valide local** : 242 tests, parse des 2 agents.
- **Valide DSS (RUN TEST du tool, par l'user)** : la resolution marche (blanchard, Algerie Telecom) ;

## Run 2 - Branchement `attribute_lookup` (ORCHESTRATEUR built-in) + durcissement + multi-table

### Objectif
Finir l'histoire du resolver : conseil multi-agents pour decider OU brancher `attribute_lookup`,
puis implementer, durcir, et preparer le recoll DSS. Justesse d'abord, rapidite ensuite.

### Fait
- **Conseil multi-agents (Workflow)** : moi = PDG technique + 4 conseillers (archi LangGraph/Dataiku,
  ingenieur SQL/perf Postgres, expert ACI/tool-design, sceptique) + 2 contradicteurs adversariaux.
  Verdict **4/4 = brancher dans l'ORCHESTRATEUR** ; la branche sous-agent ne survit pas (dominee
  latence + recout contrat `KNOWN_*`).
- **Branchement = built-in de l'orchestrateur** (`OWIsMind_orchestrator.py`) : spec dans
  `build_tool_specs`, dispatch inline dans `node_tools` (comme `show_table`/`current_date`) ->
  **zero contrat `KNOWN_*` touche**, **sous-agent INCHANGE**. Helpers `_get_tool` (fallback par nom
  si `LOOKUP_TOOL_ID` vide), `_run_lookup` (ne leve jamais -> repli gracieux vers le specialiste),
  `_extract_lookup_output`, `_lookup_tool_output`, `_lookup_evidence_item`.
- **Provenance Evidence** : le built-in emet un subspan `semantic-model-query` (le vrai canal lu par
  le backend ; `state['captured']` etait vestigial, jamais relu). `state['latest']` non touche
  (pas d'artefact pour un lookup, evite d'ecraser le resultat du specialiste en turn mixte).
- **Routage** : descripteur etroit + regle prompt ("pour `<attribut> de <entite nommee>` -> lookup ;
  pour tout chiffre calcule/liste -> specialiste"). Firewall d'honnetete : `not_found`/`suggestions`
  ne fait jamais affirmer l'absence des donnees.
- **Multi-table generique** (exigence user) : le modele passe un **domaine logique**, l'orchestrateur
  resout vers la table whitelistee via le registre (`lookup_domains()` lit `lookup_dataset` /
  `lookup_catalog` par capability). Rule #3/#4 respectee (le nom de table ne quitte pas le serveur).
  Le tool accepte `dataset`/`catalog` en entree (defaut = config). Un 2e agent = une ligne registre.
- **SQL lisible** (demande user) : `build_search_sql` passe de **18 `OR ... ILIKE`** a **UN seul
  `ILIKE`** sur `concat_ws` de toutes les colonnes texte. Accents geres a la requete par `translate`,
  **donnees jamais modifiees**. Pliage **symetrique** needle/colonne (meme map `translate`).
- **Accents** : user a d'abord choisi l'extension `unaccent` (SQL plus court) PUIS a refuse de toucher
  la base de prod -> **revenu a `translate` (zero install)**. Branche `unaccent`/flag `USE_UNACCENT`
  ensuite **retiree** (zero code mort) ; fonction renommee `unaccent_lower_sql` -> **`accent_fold_sql`**.
- **Durcissement tool** : flags `rows_capped` (LIMIT atteint -> echantillon) et `multi_column` (terme
  dans >1 colonne -> ambigu), garde needle court (`MIN_NEEDLE_CHARS=2`), cache TTL borne
  (`_cache_key` inclut le dataset), message `not_found` adouci (jamais "absent des donnees"), garde
  "rows mais rien de reconfirme + aucun attribut -> not_found".
- **Revue adversariale du diff** (Workflow, 3 relecteurs + verif) : **0 critical / 0 high**, **2 LOW
  corriges** (pliage accents symetrique pour glyphes hors-Latin-1 ; garde `found` vide).
- **Nettoyage commentaires** (demande user) : commentaires = explications techniques pures, **zero
  empreinte IA** (retire "technical board", "board decision", "we do not", justifications, dates de
  decision) dans le tool + l'orchestrateur + doc.
- **`attribute_lookup_tool.py` reecrit propre** ; **`Value_Catalog` + fallback alias/`suggestions`
  GARDES** (decision user finale, voir Decisions).

### Decisions
- **Brancher dans l'orchestrateur** (built-in), pas le sous-agent : latence + zero contrat gele +
  sous-agent intact (1 seul agent a recoller).
- **Garder le full-text `ILIKE` sur le fact** (couverture max, autonome) plutot que dictionary-first
  (value_index) : ce dernier reduit la couverture (colonnes indexees seulement) et n'a pas de lien de
  ligne (la lecture "X de <entite>" retombe sur le fact). Perf 50 users = seq-scan borne (LIMIT +
  timeout + cache) ; index `pg_trgm` = a PROPOSER au DBA, jamais cree par l'agent (NO INSTALL).
- **Accents par `translate` (zero install)**, pas `unaccent` : user refuse de toucher la base de prod.
- **GARDER le Value_Catalog + les suggestions** : revirement apres une tentative de suppression non
  validee. L'user veut absolument que ca suggere quand on ne trouve rien.
- **Vieux tool `Drive_Revenues_resolve_filter_value`** : a SUPPRIMER cote DSS (jamais appele par
  aucun agent ; charge le catalog 170K en pandas-RAM = danger). `resolve_filter_value` reste un
  **label de timeline** dans le code, pas un appel de tool -> supprimer l'objet DSS ne casse rien.
- **Datasets de connaissance** : `profile` + `value_index` = indispensables au SOUS-AGENT (comprendre
  + resoudre les noms) ; `Value_Catalog` = optionnel (seulement le fallback alias du tool, garde).
