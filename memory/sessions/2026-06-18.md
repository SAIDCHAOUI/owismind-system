# Session 2026-06-18 - Resolver rapide `attribute_lookup` + suppression totale de `dataset_lookup`

## Objectif
Deep-dive de l'archi tools + SQL (demande user), puis regler le vrai point de douleur DSS : les
questions simples ("quel est l'account manager de X ?") sont LENTES (semantic model ~80s) et echouent
("pas dans les donnees" alors que la colonne existe), et l'orthographe des colonnes casse tout.
Construire un vrai resolver rapide, le rendre universel, et retirer l'ancien `dataset_lookup`.

## Fait

  stdlib + `dataiku`, env 3.9/3.11 OK). Comportement = recherche "Whole data" Dataiku :
  - `ILIKE` sur **toutes les colonnes texte** du fact (insensible casse + accents, needle accent-strippe).
  - Sortie **`found_in`** = la/les colonne(s) ou le terme apparait + sa/ses valeur(s) exacte(s) (le
    "resolver" : ou est la valeur + orthographe exacte). `attributes` (optionnel) -> valeurs des

    `attribute_unknown` / `bad_input`.
  - **Fallback alias** optionnel sur `DRIVE_Revenues_Value_Catalog` (suggestions "vouliez-vous dire")
    quand rien ne matche ; le catalog est OPTIONNEL (un catalog absent ne casse rien).
  - **Securite** : read-only (`statement_timeout` + `transaction_read_only`), `LIMIT` partout, rien
    charge en RAM ; seuls des noms de colonnes valides du schema live entrent dans le SQL.
  - **Zero nom de colonne metier en dur** (regle P3) : colonnes lues du schema live ; priorite par
    signaux generiques du catalog (`is_alias`, `frequency`, ordre des domaines). Reutilisable sur un

## Decisions
- **Resolver = recherche SQL full-text directe sur le fact (ILIKE), PAS via le catalog.** Cause : le
  catalog n'indexe pas certaines colonnes (`account_manager`) -> "blanchard" introuvable via catalog.
  Le full-text couvre TOUTES les colonnes et trouve "blanchard", "indirect", "Telesat", etc.
- **SQL gagne sur le pandas-en-RAM** a 170K lignes (catalog) : charger 170K x 22 = ~0.5-1 Go RAM +
  fuzzy CPU lourd sous 50 users. "tool vs inline" n'est PAS un debat de charge SQL (un tool qui
  requete fait le meme SQL) ; le vrai levier = ou se fait le matching (base vs RAM) -> base.
- **Sortie minimale "resolver"** : `found_in` par defaut ; les valeurs des autres colonnes seulement si
  demandees. (L'user a iterativement recadre : pas une fiche complete verbeuse, juste ou + valeur.)
- **Ambiguite = noms vraiment differents seulement.** Meme nom vu comme compte ET groupe -> meme
  entite -> on prend le plus precis et on repond (plus de fausse ambiguite).
- **`dataset_lookup` supprime completement** (zero code mort, exigence user explicite), pas juste un flag.
- **Branchement DIFFERE** : la conception finale du tool + le point de branchement (sous-agent vs
  orchestrateur) seront tranches la prochaine session via un workflow multi-agents (user = decideur).

## Valide / non valide

- **Valide DSS (RUN TEST du tool, par l'user)** : la resolution marche (blanchard, Algerie Telecom) ;
