# Session 2026-06-29 - Benchmark phase finale : juge contextuel + override humain + consultation dans le plugin

## Objectif
Perfectionner le systeme de benchmark : (A) juge plus strict mais contextuel + colonne commentaire ;
(B) humain dans la boucle (revoir/override le verdict du juge) ; (C) integrer la CONSULTATION des
resultats dans la webapp plugin owismind (dropdown d'agent, pour tous) + cabler un benchmark a une
fiche d'agent (table + validation de schema). Generaliser a n'importe quel agent Dataiku.

## Decisions (brainstorm + retours en cours de route)
- **Juge = autorite contextuelle** : l'ancre objective redevient un SIGNAL (un HIT confirme, un MISS
  ne force plus "faux", le juge tranche) ; parsing des ordres de grandeur ; la note humaine = contrat
  de severite ; colonne `judge_comment` concise. (Spec = `docs/superpowers/specs/2026-06-29-benchmark-final-phase-design.md`.)
- **1 table `scored` par agent** : le plugin lit cette table en cross-projet et recalcule la
  restitution (agregateur dedie `benchmark_view/aggregate.py`, pas de copie de `scoring.py`).
- **Override = UPDATE dans `scored`** (colonnes `human_*`). Robuste : la table empile l'historique par
  `run_id`, donc un override survit aux runs suivants (seul re-juger le MEME run_id l'efface).
- **Plugin = CONSULTATION uniquement** (decision user en cours de session) : AUCUN launcher dans le
  plugin ; le lancement (config + golden + suggestions + run) reste sur les webapps LAB. L'override
  admin par question reste dans le plugin ("laisser tel quel").
- **Reproduction NATIVE du design LAB results** dans la consultation du plugin (exigence user : "pas
  d'iframe"). Port du CSS de la webapp LAB `results` mappe sur les tokens semantiques du plugin.

## Run 2 (apres-midi) - Consultation benchmark en PLEINE LARGEUR (parite webapp LAB results) + Q/R conseil sur la taille d'un benchmark

### Objectif
Retour user : l'onglet Benchmark du plugin rend "confine au milieu" alors que la webapp LAB `results`
rend pleine largeur ("le jour et la nuit"). Vouloir le MEME rendu que le LAB, proprement, sans casser
les autres vues. Puis 2 questions de fond (override admin : quoi/quelle table ? ; combien de questions
pour un benchmark credible ?).

### Fait (frontend SEUL, repo only)
- **Diagnostic** : la consultation est deja la reproduction native du LAB ; le seul ecart = le wrapper
  partage `PageShell` plafonne le contenu a **880px centre** (`.page-wrap max-width:880px; margin:0 auto`,
  ou 1080px en `wide`). Le LAB `results` n'a aucun plafond (`.content flex:1`, aside 360px, padding 40px).
- **`components/pages/PageShell.vue`** : nouveau prop **opt-in `fluid`** -> `.page-wrap--fluid
  { max-width:none; margin:0; padding: var(--s-8) var(--s-8) var(--s-10) }`. Les modes existants (880
  defaut / 1080 `wide`) INCHANGES. Les autres vues ne passent pas le prop => zero impact (la garantie
  demandee par l'user : on n'elargit pas le conteneur pour tout le monde, on ajoute une variante choisie
  page par page).
- **`views/BenchmarkSuggestView.vue`** : passe `fluid` a PageShell ; aside `consult-aside` 320 -> **360px**
  (parite LAB) ; KPIs passent a 3 colonnes des **1280px** (au lieu de 1080, comme le LAB) ; la section
  "Suggerer" (formulaires) gagne `.bench-section--narrow { max-width:880px }` pour rester une colonne
  lisible pendant que la consultation au-dessus est pleine largeur.
- **Build** : compile-check Vite OK ; **zip DEV regenere** `owismind_dev-upload.zip` (78 entrees, entry
  **`index-CzZWTpbS.js`**), invariants `--check` + build OK. **PROD intacte** (resource/body.html prod non
  touches ; seuls les 2 fichiers source modifies). 0 tiret cadratin/demi-cadratin sur les 2 fichiers.

### Reponses de fond (pas de code)
- **Override admin depuis le plugin** : bouton Mark correct/incorrect/Clear -> `store.submitOverride` ->
  POST `/owismind-api/admin/benchmark/override` -> `lab_io.write_override` = **UPDATE parametre** de la
  table **SCORED du LAB en cross-projet** (resolue serveur depuis le bloc `benchmark {connection, table}`
  de la fiche d'agent ; jamais envoyee par le client). Ecrit `human_verdict / human_correct / human_comment
  / reviewed_by / reviewed_at` sur la ligne (run_id, question_id, agent_key, mode), COMMIT, statement_timeout.
  Admin-only + bloque en impersonation (route WRITE). Clear remet a NULL -> retombe sur le juge. Survit aux
  runs (scored empile par run_id). MEME table que l'onglet Review/override du launcher LAB (`dss.write_override`).
- **Taille d'un benchmark credible** : une question pese ~`100/n` points. <30 = bruit ; **cible credible
  ~100-150** questions stratifiees (mix calque sur `docs/questions_asked.md`, ~10-15 par bucket capacite x
  intent x difficulte, + hors-sujet + garde-fous de regression) ; robuste/vitrine 250-300+. Attention : la
  stat est PAR CONFIG (n par mode, pas au total) -> Claude (cher) sur un sous-ensemble noyau ; surveiller le
  taux `needs_review` (bruit du juge). Qualite du golden > quantite.
