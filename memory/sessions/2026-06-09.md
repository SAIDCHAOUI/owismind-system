## Run 4 - Fix bug remonté EN DSS : scroll vs navigation de versions (L032 item 8) ✅ DSS
- **Objectif** : après navigation vers l'ancienne branche, le bouton « revenir à la dernière version » semblait disparaître (il fallait reload).
- **Cause racine** (debug systématique) : `ChatThread` re-pinnait le scroll en bas via `watch(() => props.turns, repin)` + `watch(() => props.turns.length, repin)` ; `turns` = computed (nouveau tableau + longueur change à chaque navigation) → **scroll forcé en bas à chaque navigation** → flèches du point de branche hors écran sur une branche longue.
- **Fait** : scroll en bas **seulement** sur `chat.activeSessionId` (switch), `chat.exchanges.length` (nouvel échange), et streaming **gated sur `chat.sending`**. Plus jamais sur une navigation de version.
- **Preuve empirique** (dev server + Chrome DevTools, avant/après) : corrigé `atBottom=false` (flèche visible), bug réintroduit `atBottom=true` (scroll forcé) → causation prouvée (`frontend/.scroll-bug-screens/`).
- **Validé DSS** : user « tout fonctionne à merveille ».

## Run 6 - Bouton « Stop génération » (arrêt d'une réponse en cours) ✅ VALIDÉ DSS
**Objectif** : permettre d'arrêter la génération en cours (mauvais prompt) ; le prompt + la réponse partielle restent stockés ; le tour suivant est un nouvel échange ; éditer = branche. Via brainstorming → spec (`docs/superpowers/specs/2026-06-09-stop-generation-design.md`) → implémentation (user a dit « go »).
**Constat doc Dataiku officielle** : `completion.execute_streamed()` = **itérateur** de chunks ; **aucune méthode `stop()`/`cancel()`/`close()`** documentée → arrêter = **cesser d'itérer** (`break`). C'est déjà le mécanisme `stream_manager` pour `timeout`/`abandoned` (break + persistance du partiel). Donc seul l'**arrêt déclenché par l'utilisateur** manquait.
**Fait (backend)** : `stream_manager` → `start_run` init `stop_requested:False` ; `request_stop(run_id,user_id)` **owner-scopé** ; `_stop_reason` renvoie `"stopped"` en priorité ; worker émet un event terminal **`stopped`** (≠ `error`, pas de `state["error"]`) après le `final_answer` partiel (persistance phase 2 inchangée). `routes.py` → **`POST /chat/stop {run_id}`** (401/400 `invalid_run_id`/404 `run_not_found`, borné par `_MAX_RUN_ID_LENGTH`) + inventaire docstring.
**Fait (frontend)** : `backend.js` `stopChat()` ; `useChatStream` callback `onRunId` (le polling **continue** après stop pour rendre `final_answer`+`stopped`) ; `chat.js` `activeRunId`+`stopPending`+`stopGeneration()` (best-effort, ignore `run_not_found`, gère la course « stop avant run_id ») ; `timelineModel` case `stopped`→`status='stopped'` ; `PromptBar` bouton ↑→■ pendant `sending` (+icône `stop` dans `icons.js`) ; `MessageAgent` marqueur « ⏹ Génération arrêtée » sous le partiel + placeholder « Réponse interrompue » si vide ; clés i18n fr/en dans `extra.js`.
**Décisions (user)** : tour arrêté = **tour normal** dans le contexte (D1) ; rendu = **marqueur discret + placeholder si vide** (D2). **Aucun changement de schéma** (marqueur live-only ; au reload, partiel tel quel ou placeholder si vide).
