## Run 1 (nuit déléguée) — Construction du système

## Objectif
Session entièrement déléguée par l'user (parti dormir, « tu prends toutes les décisions ») :
construire dans le repo un système agentique **générique** — orchestrateur + sous-agent expert de
N'IMPORTE QUEL dataset — qui comprend les données en profondeur, génère son propre SQL et répond
juste, sans dépendre du semantic model visuel (boîte noire). Livrable exigé : dossier complet
(agents + recettes + tools + README d'implémentation détaillé), focus RÉALISATION.

## Fait
- **Recherche préalable (workflow 6 agents, ~367k tokens)** : doc officielle Dataiku (code agents,
  LLM Mesh, tools, traces), semantic model 14.4 (API Python existe : `create_semantic_model`, spec
  JSON complète), SQLExecutor2/read-only/`dataiku.recipe`, état de l'art NL2SQL production
  (Uber/Pinterest/LinkedIn/Snowflake/dbt), taxonomie complète des 817 questions réelles, critique
  d'architecture des 2 agents existants. Conclusions clés : templates déterministes + couche
  sémantique ≫ SQL libre (98-100 % vs 84-90 %, dbt 2026) ; le corpus est dominé par les follow-ups
  elliptiques (~22 %) et les agrégations filtrées (~22 %) ; ~14 % de questions mono-dataset
  NON-revenu (tickets/CX/opps) ; toutes les engueulades = « répondu sans router » (confirme v2.4).
- **`dataiku-agents/recipes/profile_dataset_recipe.py`** : recette Flow générique → dataset
  `<X>_profile` `{key, payload JSON}`. Passe A déterministe (schéma, types, nulls, distincts, enums
  verbatim ≤50 valeurs, samples, stats, détection format temporel date/yyyy_mm_dd_str/yyyy_mm_str/
  yyyymm_int/year_int) + Passe B LLM Mesh (descriptions fr/en, rôles, synonymes, métriques
  {agg,column,format,unit}, élection colonne scénario + défauts, display pairs id↔label), validation
  déterministe `validate_enrichment`, **overrides humains** via 2ᵉ input éditable `{key,field,value}`
  appliqués EN DERNIER (jamais écrasés), `comment` UI récupéré comme description de départ.
- **`dataiku-agents/recipes/build_value_index_recipe.py`** : index de valeurs générique →
  `<X>_value_index` `{column_name, value, value_norm, occurrences}` (norm = minuscules sans accents,
  contrat FROZEN partagé avec l'agent), sélection auto des colonnes groundables (skip numériques/
  dates/free-text/ids longs, cap 20k valeurs/colonne), INCLUDE/EXCLUDE manuels.
- **`dataiku-agents/agents/dataset_expert_agent.py` (~1500 l.)** : sous-agent générique standalone.
  UNDERSTAND (1 LLM JSON, prompt + schéma JSON **générés du profil** : métriques, scénarios verbatim,
  axes+synonymes, règles périodes, terms, round-trip « VALEUR (Colonne) ») → validation/dégradation
  pure contre le profil (stopwords DÉRIVÉS du profil, P3 respecté ; compare_scenarios mono-valeur
  préfixe le scénario défaut → généralise ACTUALS-vs-BUDGET sans dur-codage) → RESOLVE par SQL sur
  l'index (passe exacte groupée IN, passe fuzzy LIKE + ranking difflib, repli slice 5000, politique
  d'ambiguïté portée de v2 : colonne préférée/éviction exacte/priorité par `ambiguity_priority` puis
  distinct_count) → BUILD SQL : **9 intents → templates déterministes** (total, breakdown, top_n,
  share_of_total fenêtré, compare_scenarios pivot CASE+delta+delta_pct, compare_periods pivot,
  trend bucket mensuel par format, list_values, count_distinct ; prédicats de période par format
  temporel ; display pairs `MAX(label)` ; identifiants quotés validés, littéraux échappés) ; intent
  `custom` (longue traîne) → LLM SQL contraint par dataset card + **GARDE-FOU** (un seul SELECT,
  table whitelistée avec variantes+CTE, mots-clés interdits, LIMIT forcé/cappé) + EXPLAIN à blanc +
  **boucle de réparation ≤2** (erreur DB réinjectée) → EXECUTE SQLExecutor2 `query_to_iter` (repli
  pandas) avec `SET LOCAL statement_timeout/transaction_read_only` (pattern L045) → RENDER : table
  par code (formats pilotés par le profil : amount/unit/percent/count — plus de sniffing métier),
  headline LLM **vérifiée chiffre par chiffre**, no_data honnête + hints métadonnées (scénarios
  dispo, couverture temporelle) ; intent `about_data` = carte du dataset 100 % profil, 0 SQL.
  Events : même dialecte (blockIds `KNOWN_BLOCK_IDS`, tools `KNOWN_TOOL_NAMES`, `AGENT_RESULT`
  +`attempts`) ; spans `semantic-model-query` au contrat gelé avec **success VÉRIDIQUE** par
  tentative. Profil lu au runtime avec cache TTL 600 s (autonomie : re-run recette = agent à jour).
- **`dataiku-agents/agents/orchestrator_agent.py` (v3.0)** : copie du v2.4 (pare-feu d'honnêteté
  intact) + **EXECUTE PARALLÈLE** (≥2 étapes agent → ThreadPoolExecutor borné `MAX_PARALLEL_AGENTS=3`,
  workers streament et poussent les events dans une queue, le générateur principal les re-yield live ;
  spans/usage/tagging SQL post-hoc sur le thread principal ; deadline 600 s → étapes marquées error

- **`dataiku-agents/tests/` : 110 unittest DSS-free verts** (2 skips pandas couverts en DSS) :
  profiler (formats temporels, enrichment, overrides, rôles, sélection index), expert (profil,

- **`dataiku-agents/README.md`** : guide d'implémentation détaillé en français (architecture, pas à

## Décisions (prises en délégation)
- **SQL code-owned plutôt que semantic model** (même pilotable par API en 14.4) : contrôle total des
  prompts, capture Evidence exacte, `success` réel, boucle de réparation possible, testable offline.
  Le semantic model reste documenté comme alternative ; rollback = flags du registre.
- **Aucun tool DSS managé requis** : resolver et SQL inlinés dans l'agent (SQL direct sur l'index et
  le dataset) → déployer un dataset = 2 recettes + 1 Code Agent, zéro tool à câbler.
- **Expertise = artefacts du Flow** (profil + index) relus par l'humain via dataset éditable
  d'overrides — les données ne sortent jamais de DSS, le repo ne contient AUCUNE valeur métier
  (fixtures synthétiques dans les tests).
- **Orchestrateur v3 collé dans le Code Agent EXISTANT** (même id) → webapp intacte.
- Modèles : profilage = le plus fort (coût unique) ; UNDERSTAND/headline → Flash après validation ;
  SQLGEN (custom uniquement) = fort. Ids par défaut = le Gemini 2.5 Pro connu de l'instance.
