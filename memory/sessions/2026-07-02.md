## Decisions
- **`source` sur show_chart/show_table/show_kpi : implemente puis RETIRE** (revert complet). La
  revue finale a prouve que le plugin ne lie qu'UN `result_block` a TOUS les artefacts d'un echange
  (`api/routes.py:804-809` : meta["result"] unique -> build_chart_payload pour chaque artefact) ->
  dans un tour a 2 specialistes, le 2e graphique se serait affiche VIDE (pire que l'actuel "skipped"
  avec message au modele). Le vrai fix = artefact tagge `agent_key` + binding par artefact dans
  `/evidence/meta` = chantier PLUGIN (zip + restart) -> hors scope sans feu vert user, documente
  comme suite possible.
- **AGENT_DONE relay du SQL lookup (stop-safety) : SKIP** (finding confirme mais benefice marginal :
  fenetre de perte etroite, et un run stoppe n'a pas de reponse finale de toute facon ; risque UX
  timeline d'un AGENT_DONE sans agent).
- Spans multi-SQL : garder success=True (un None s'affiche en echec cote `_normalize_sql_item` ;
  on n'a aucun signal par-SQL), ne retirer QUE le row_count mensonger.
- Marqueur d'ambiguite : generique plutot que 2 variants (impossible de savoir cote agent si les
  colonnes sont des colonnes d'offre sans hardcoder du metier, P3) ; le cerveau semantique porte la
  distinction (il connait ses colonnes).
- Moteur tickets (copie du moteur revenus) : PAS patche (hors scope explicite de l'audit) -> les
  fixes moteur sont a PORTER dans `OWISMIND_DEV_CSSO_Trouble_Tickets_Expert.py` apres validation
  DSS des fixes revenus (meme discipline que la promotion PROD).
